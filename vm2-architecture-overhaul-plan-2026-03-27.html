These aren't seven separate bugs — they're symptoms of one missing system: a self-documenting deliverable architecture where every output preserves its own context and registers itself in a single index.

2. Architecture Decisions

2A. portal.html Is the Deliverables Index (Final)

portal.html is the one and only deliverables index. This is non-negotiable and permanent.

  • index.html becomes a permanent, immutable HTML redirect — <meta http-equiv="refresh" content="0;url=portal.html"> plus a JS fallback
  • index.html must NEVER be edited by any session — it contains zero content
  • nginx.conf already enforces: location = /index.html { return 302 /portal.html; }
  • Memory and all skills updated to enforce this
Why Both?
The nginx redirect handles HTTP traffic. The HTML redirect handles direct file opens and edge cases where nginx isn't in the path. Belt and suspenders.

2B. The Context Slug System

Every deliverable HTML gets a <section id="vm2-context"> block containing structured JSON and human-readable text. This is the continuity mechanism.

This is embedded as both:

  2. A visible, collapsible "Session Context" panel at the bottom of the page — human-readable for Varun

2C. The "Continue This Work" UI Block

At the bottom of every deliverable, a styled panel with three action buttons:

  1. Copy Follow-Up Prompt — copies to clipboard a pre-written prompt that points the next session to the context slug URL with instructions to read all prior reasoning, findings, dead ends, and open questions
  2. Email to VM2 — mailto: link to vm2.viru@gmail.com with the subject and body pre-filled. The Gmail task pipeline picks this up and creates a new session.
  3. Create Todoist Follow-Up — POSTs to /api/todoist/tasks (the Railway nginx proxy) to create a labeled task with the context URL

2D. Version Tracking in portal.html

When a deliverable is updated:

  • New file uses convention: slug-YYYY-MM-DD-v2.html (or v3, etc.)
  • The new version's context slug references all prior versions in the prior_versions array
  • portal.html entry shows version badges with links to v1, v2, v3
  • Each version's context preserves the full chain of reasoning

2E. Session Isolation (Practical Reality)

  • Cron jobs use run_subagent internally — these are NOT revisitable sessions
  • The Gmail task pipeline creates separate Perplexity Computer sessions — this is the preferred path for trackable work
  • The Auto-Helper "Task VM2" email buttons are the right mechanism — they trigger the Gmail pipeline which creates proper sessions
  • For cron-produced deliverables (daily briefs, SAM scans), the context slug IS the continuity mechanism — paste the Railway URL into a new session to continue
Every deliverable is self-contained with enough context to restart from where it left off. Sessions are ephemeral. Deliverables are permanent.

4. Token Economics

The context slug adds 3–8K tokens per deliverable. Here's the cost analysis showing why it pays for itself on the first follow-up.

ScenarioInput TokensOutput TokensTotal Cost
Follow-up WITH context slug ~42K ($0.13) ~15K ($0.23) ~$0.36
Follow-up WITHOUT context (from scratch) ~30K ($0.09) ~30K ($0.45) ~$0.54
Loading 3 related deliverables ~51K ($0.15) ~15K ($0.23) ~$0.38
Note
Based on Claude Sonnet 4.6 pricing ($3/$15 per 1M tokens). The context slug costs ~3–4 cents to read but saves 2–3× in redundant output and tool calls. The real ROI is quality: a follow-up session that reads the context slug doesn't repeat dead ends or miss prior findings.
