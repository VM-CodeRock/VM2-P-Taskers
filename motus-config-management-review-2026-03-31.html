Tools and Platform Observations

  • Terraform + AWS CodeCommit: Infrastructure is versioned in CodeCommit (Fmcsa-registration-terrascripts). The use of per-environment .tfvars and .config files is a mature pattern that enables environment parity.
  • Jenkins CI/CD: Terraform Plan and Apply jobs are Jenkins-managed. Dev auto-applies on merge; INT, STG, TRAIN, PROD require manual release execution — appropriate access control for production-tier environments.
  • AWS ECS + ECR: Application deployments are containerized. ECS deployment verification is a required post-deployment step — this is correctly documented.
  • AWS Secrets Manager: Secrets are runtime-retrieved; a centralized model means lower environments pull shared secrets from dev. This is functional but carries a minor risk: if the dev environment is compromised or misconfigured, dependent environments may be affected.
  • Confluence: Used for knowledge management and version tracking. Effective for a small team but lacks the formal workflow enforcement of a dedicated ITSM or CM tool.

4. Strengths
What's Working Well
The team has established several practices that demonstrate operational maturity and align with DevSecOps best practices for federal systems.
  • Infrastructure as Code (Terraform): Codifying all infrastructure in Terraform, stored in CodeCommit, with state managed via S3 + DynamoDB, is the right approach for a federal system. It enables reproducibility, peer review, and audit capability.
  • Single Codebase, Environment Parameterization: Using one Terraform codebase with per-environment .tfvars files reduces configuration drift, ensures infrastructure parity, and simplifies maintenance. This is a mature pattern.
  • Protected Master Branch with PR Workflow: No direct commits to master; every infrastructure change goes through a feature branch → Terraform Plan → PR with plan link → approval → merge. This is a solid change control mechanism for infrastructure.
  • Secrets Externalized: Secrets are stored in AWS Secrets Manager and retrieved at runtime. No secrets embedded in code or Terraform files. This meets a core NIST SP 800-53 requirement (SC-28).
  • Automatic Rollback on Failure: ECS automatic rollback is enabled for failed deployments — reduces manual intervention and limits blast radius from a failed release.
  • CloudWatch Logging: Container and deployment logs in CloudWatch provide a basic audit trail for deployment events and application errors.
  • Daily Version Visibility: Robert's practice of posting daily version snapshots to the team channel, captured in Confluence by Kairine, creates operational visibility that many teams lack. This is a genuine differentiator.
  • Multi-Stage Environment Pipeline: Five distinct environments (Dev → INT → Staging → Training → Prod) with access-controlled promotions to Staging and above shows a mature release pipeline design for a federal system.
5. Gaps & Recommendations

Gap 1: No Formal Configuration Management Plan Document

What exists is an operational runbook for Terraform and a version log. A federal system CM Plan should be a discrete, versioned artifact covering: purpose and scope, CM roles and responsibilities, configuration identification (CI numbering), change control process, configuration status accounting, configuration audits, and subcontractor/vendor oversight. NIST SP 800-128 (Guide for Security-Focused CM) requires this explicitly for federal information systems.

Recommendation
Draft a formal CM Plan document (even a concise 5–8 page document) that defines: CM manager/roles, CI registry, CCB process for major changes, release authority matrix, and audit cadence. This document is often required for ATOs and contract deliverables.

Gap 2: Change Control Board (CCB) / Release Authority Not Defined

The Terraform PR workflow provides change control for infrastructure, but there is no defined authority or process for: approving application releases from Staging to Production, handling emergency hotfixes outside the normal process, or escalation paths for deployment failures. The 36-day Production lag may be deliberate — but there is no documented evidence of who authorized this cadence.

Recommendation
Define a Release Authority Matrix: who approves Staging → Production promotions, what constitutes an emergency release, minimum soak time in Staging before Production, and required sign-offs (e.g., QA lead, Security, FMCSA COR).

Gap 3: No Configuration Baseline or Security Configuration Standards

There is no documented security configuration baseline for the system — no STIG/CIS benchmark application, no documented deviation from baseline, no baseline versioning. This is typically required for FedRAMP and FISMA-covered systems.

Recommendation
Establish a documented configuration baseline capturing approved security configurations for ECS, EC2/Fargate, RDS, S3, IAM, and application containers. Track deviations from baseline as formal Configuration Change Requests (CCRs).

Gap 4: No Software Bill of Materials (SBOM) or Dependency Tracking

There is no documentation of third-party dependencies, open-source libraries, or their versions. For a federal system, this creates supply chain risk (per EO 14028) and makes vulnerability management reactive rather than proactive.

Recommendation
Integrate automated SBOM generation into the CI/CD pipeline (e.g., Syft, Grype, or AWS Inspector). Publish SBOM artifacts alongside each release. This is increasingly required in federal contracts and aligns with NIST SSDF practices.

Gap 5: Version Reporting Script Has Data Quality Issues

The March 26 snapshot shows all environments reporting Web version "18.3.1" — inconsistent with the "1.13.x" format seen in all other snapshots. The INT environment in the March 30 snapshot shows Web "1.2.28" instead of the expected "1.13.x" format. These anomalies indicate the Python version-reporting script may have data quality gaps (e.g., scraping the wrong field in certain conditions).

Recommendation
Add validation logic to the get_versions.py script: version format validation, alerts on anomalous values (e.g., if web version doesn't match expected pattern), and a structured JSON output that can be compared against a known schema. Store version history as structured data, not just Confluence snapshots.

Gap 6: No Documented Production Deployment Cadence or Release Schedule

The 36+ day Production lag is unexplained. There is no documented release schedule, no documented hold reason, and no release window policy visible in the documentation. For a federal system with business-critical operations, the absence of a documented release cadence is a compliance and operational risk.

Recommendation
Document the Production deployment schedule (e.g., bi-weekly, monthly), required staging soak time, go/no-go criteria, and who initiates the release pipeline for Production. Add this to the CM Plan.

Gap 7: Centralized Secrets Model Risk

Environments sharing the same secret values pull from the dev environment's AWS Secrets Manager. While this reduces duplication, it creates a dependency: if the dev account is decommissioned, incorrectly configured, or access to it is revoked, dependent environments may fail to retrieve shared secrets at runtime. This model also means that dev-environment IAM changes can impact upstream environments.

Recommendation
Document which secrets are shared vs. environment-specific. Evaluate whether the centralized model should be replaced with environment-specific secrets for all Staging, Training, and Production tiers. At minimum, document this dependency as a known architectural decision with its accepted risk.
6. Risk Assessment
Production Version Lag — 36 Days
Production running Web 1.13.2 / Server 1.2.23 / Migration 352, while Dev/INT are at 1.13.5 / 1.2.28 / 379. If any of the 27 DB migrations contain security patches or critical fixes, Production users are exposed. No documented justification for the delay exists.
No Formal ATO-Ready CM Documentation
Federal systems under FISMA require a CM Plan as part of the System Security Plan (SSP). The current documentation is insufficient to satisfy an ATO review without supplemental narrative. This may create audit exposure if FMCSA reviews the contract deliverables.
INT Environment Web Version Anomaly
INT reporting Web "1.2.28" instead of "1.13.5" in the latest snapshot is unexplained. If this is a real version discrepancy (not a script error), it means INT may be running a non-standard application build that doesn't reflect what Staging and Production will run.
Seeder Version Drift (STG/TRN vs. Dev/INT)
Staging and Training at Seeder 95 vs. Dev/INT at Seeder 91. If seeder scripts modify or populate test data, different seeder versions may cause test results in Staging to not accurately reflect Development behavior, potentially masking issues before they reach Production.
Version Script Data Quality Issues
The Mar 26 snapshot showed all environments at "18.3.1" (Node.js version, not app version) — a silent data quality failure. If the team is relying on this script for operational decisions, incorrect data could go unnoticed without validation logic.
Shared Secrets Model (Dev → Upstream Dependency)
INT and potentially other environments sharing secrets from the dev AWS account creates an undesired coupling between the lowest-trust environment and production-grade environments. Account-level incidents in dev (e.g., compromised credentials) could affect upstream runtime behavior.
No SBOM / Supply Chain Visibility
Open-source dependencies are not tracked or inventoried. While likely low-frequency risk in the short term, federal requirements (EO 14028, NIST SSDF) are trending toward mandatory SBOM production for government systems. Early adoption reduces future remediation cost.
Confluence as Single Source of Truth
Version history lives in Confluence snapshots, which are human-generated and not machine-queryable. If Confluence access is lost, version history is inaccessible. Storing structured version data in a persistent data store (S3, database) provides better durability.

Context Note
Federal systems operating under FISMA are required to implement NIST SP 800-53 controls. The CM control family (CM-1 through CM-11) defines the minimum requirements. The current documentation satisfies approximately CM-2 (partially), CM-5 (access controls via Jenkins), and CM-9 (configuration management plan — but only the operational portion). The strategic and governance elements (CCB, baselines, audits, inventory) represent the most significant gaps and should be the priority for the next documentation sprint.
