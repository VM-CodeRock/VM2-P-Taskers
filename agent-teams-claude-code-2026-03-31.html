Agent Teams in Claude Code

Multi-Agent Orchestration for Federal IT Consulting
March 31, 2026 · Prepared for Varun Malhotra · Changeis Inc.
1. Executive Summary

Anthropic's Agent Teams is a built-in multi-agent orchestration feature for Claude Code (their terminal-based agentic coding tool) that lets multiple Claude instances work in parallel on a shared task list, communicating directly with each other. It was released in experimental form in early 2026 and is available now on Claude Pro ($20/month) and Max plans.

Bottom line for Changeis: Agent Teams is genuinely powerful for parallelizable development work — especially large codebases, multi-module refactoring, and cross-layer coordination. For Varun's use cases, the highest-value applications are internal tool development (VM2 infrastructure, RAG engine) and parallel code QA workflows. It is not a replacement for Perplexity Computer as the VM2 orchestration layer — it requires hands-on terminal interaction and is not automatable via cron today.

2. What is Claude Code?

Claude Code is Anthropic's terminal-based agentic coding tool — a CLI program that runs Claude models directly in your shell, giving the AI direct access to your filesystem, terminal, and development toolchain. Unlike IDE extensions (Cursor, Copilot) that suggest code, Claude Code acts autonomously: it reads files, writes changes, runs tests, observes compiler errors, and iterates — all without a human approving each step.

Known Limitations (Experimental)
Agent Teams is currently experimental and disabled by default. Known limitations as of March 2026: no session resumption for in-process teammates (/resume breaks teammates); task status can lag (teammates may fail to mark tasks complete); one team per session; no nested teams (teammates cannot spawn sub-teams); shutdown can be slow; split-pane mode requires tmux or iTerm2 (not supported in VS Code integrated terminal or Windows Terminal).

4. Changeis Use Cases — Feasibility Analysis

Varun runs Changeis Inc., a federal IT consulting firm with an AI-augmented operational stack (VM2 on Perplexity Computer, Todoist pipeline, Railway deployment, RAG engine, SAM.gov/APFS cron monitors). Below is a detailed analysis of where Agent Teams fits — and where it doesn't.

Internal Tool Development (VM2 Infrastructure, RAG Engine)
High Feasibility

The opportunity: Changeis maintains a growing set of internal tools — the VM2 pipeline infrastructure (cron scripts, Railway config, GitHub Actions), a RAG engine for BD opportunity matching, and various integrations (SAM.gov API, APFS, Gmail). These codebases have clearly separable layers: data ingestion, matching logic, API handlers, frontend/portal UI, and deployment scripts.

How Agent Teams helps:

  • Assign one teammate to the RAG engine data ingestion layer, another to the embedding/matching logic, a third to the API endpoints — all working simultaneously
  • Cross-layer changes (e.g., adding a new opportunity field that spans ingestion → matching → portal display) can be coordinated via peer-to-peer messaging without manual handoffs
  • A dedicated "tests" teammate can write unit tests in parallel as code teammates implement features, rather than sequentially after
  • For CLAUDE.md: define the VM2 architecture, Railway deployment patterns, and Changeis coding standards once — all teammates inherit the context

Example prompt: "Create an agent team with 4 teammates. Teammate 1: extend SAM.gov ingestion to capture new opportunity fields. Teammate 2: update the RAG matching logic to score against these fields. Teammate 3: update the portal.html schema. Teammate 4: write integration tests. Coordinate via shared task list."

Best fit: This is the highest-value use case for Changeis. The VM2 codebase is modular, well-documented (CLAUDE.md can capture all VM2 conventions), and benefits significantly from parallel development.
Client Project Code Development
High Feasibility

The opportunity: Changeis delivers federal IT projects that involve multi-layer software systems — typically with distinct frontend, backend, data, and infrastructure components. These map cleanly to Agent Teams' strength: independent work streams that need to coordinate on interfaces.

How Agent Teams helps:

  • New system builds: Separate teammates own each layer (UI, API, database schema, deployment IaC). Each works without waiting on others; shared task list tracks interface contracts
  • Bug investigations: Assign teammates competing hypotheses about a production issue — they converge on the cause faster than sequential investigation
  • Feature sprints: Parallelize frontend changes, backend API changes, and test coverage for a single feature

Caveat: For client code on classified or sensitive systems, ensure your Claude Code setup routes through appropriate data handling channels (Claude API with enterprise data handling, not the consumer Pro subscription).

Automated Testing and QA Workflows
High Feasibility

The opportunity: Writing comprehensive test coverage is time-consuming and often deferred. Agent Teams can parallelize test writing across modules.

How Agent Teams helps:

  • Spawn a 4-teammate team: unit tests (Teammate A), integration tests (Teammate B), edge case/boundary tests (Teammate C), performance benchmarks (Teammate D)
  • Use TaskCompleted hooks as quality gates — if tests don't pass a coverage threshold, the hook blocks task completion and forces the teammate to improve
  • TeammateIdle hook: if a teammate finishes but coverage is below target, exit with code 2 to send it back to work
  • Particularly valuable for VM2 cron scripts where test coverage is typically low — quickly bring all monitors up to tested standards
Hooks are the killer feature here. Quality gates enforced via hooks mean you get test coverage you can trust, not just tests that were written.
Documentation Generation
Medium Feasibility

The opportunity: Changeis likely has underdocumented internal tools and client deliverable codebases. Parallel documentation generation could be valuable.

How Agent Teams helps:

  • Assign teammates to different documentation artifacts: API reference docs (Teammate A), architecture decision records (Teammate B), user/operator runbooks (Teammate C)
  • Each teammate reads relevant code/config and produces documentation in parallel
  • A reviewer teammate synthesizes and ensures consistency across docs

Why only Medium: Documentation doesn't require cross-agent coordination — a single Claude Code session can write docs for an entire codebase in one pass. Agent Teams adds cost and complexity that may not be worth it for this use case vs. a well-prompted single session. Reserve Agent Teams for when documentation spans truly independent subsystems that benefit from specialized teammates (e.g., one teammate focuses on federal compliance documentation, another on technical API docs).

System Architecture Planning
Medium Feasibility

The opportunity: Agent Teams' "debugging with competing hypotheses" pattern maps well to architecture evaluation — multiple agents can each champion different architectural approaches and argue for their merits.

How Agent Teams helps:

  • Architecture spikes: Assign teammates to prototype competing approaches (e.g., Teammate A builds a microservices approach, Teammate B builds a monolith, Teammate C builds serverless). Each produces a working proof-of-concept, then the lead synthesizes.
  • Technology evaluation: One teammate researches and tests Option A, another Option B — both produce concrete findings rather than just theoretical analysis
  • Particularly useful for VM2 infrastructure evolution decisions (e.g., whether to move crons to GitHub Actions vs. keep on Railway vs. migrate to AWS Lambda)

Why only Medium: Architecture planning is inherently a reasoning/synthesis task that Perplexity Computer already handles well. Agent Teams adds value mainly when you want actual working code prototypes evaluated, not just research.

Federal Proposal Writing (Multiple Volumes/Sections)
Low Feasibility

Why it's tempting: Federal proposals have multiple independent volumes (Technical, Management, Past Performance, Price) that seem like they'd benefit from parallel writing agents — one per volume.

Why Agent Teams is the wrong tool:

  • Agent Teams is a coding tool. Its context, tools, and workflow are optimized for code — not for Word documents, compliance matrices, or proposal narratives
  • Proposal writing requires deep federal-specific knowledge (FAR/DFARS compliance, evaluation criteria alignment, color team review standards) that Claude Code's coding-focused context doesn't support well
  • The cross-section dependencies in proposals (Technical volume must align with Management volume themes, Price must reflect staffing in Management) require a writing-focused coordination layer, not a code task list
  • File outputs would be code files (.md, .txt), not properly formatted Word documents with tables, formatting, and headers

What to use instead: Perplexity Computer remains significantly better suited for proposal work — it can reason about compliance requirements, access research, and output structured documents. Use VM2's existing research and writing capabilities for proposals.

Don't force Agent Teams onto proposal writing. It will produce lower-quality output at higher cost vs. a well-structured Perplexity Computer session with Changeis style guide context.
Integration with Existing VM2 Pipeline
Low Feasibility — Today

The appeal: Could Agent Teams be invoked from VM2 cron jobs to parallelize development tasks automatically?

Current reality:

  • Agent Teams requires an interactive terminal session. There is no programmatic API to spawn and manage an Agent Team from a cron job or external script
  • The feature is experimental and designed for developer-in-the-loop use, not fully autonomous pipelines
  • Even Claude Code's standard CLI mode (without Agent Teams) is not designed for fully headless, unattended operation in the way VM2's Railway crons work

Future potential (Medium, 12–18 months): As the feature matures and Anthropic exposes programmatic team orchestration via the Claude API, integration with VM2 becomes plausible. The pattern would be: VM2 Todoist task → trigger Claude Code Agent Team session → team executes code changes → pushes to GitHub → Railway redeploys. This is not available today.

The right mental model: Agent Teams is a tool Varun uses interactively when doing development work, not a component VM2 orchestrates automatically. Perplexity Computer remains the automation layer; Claude Code Agent Teams is the interactive developer experience layer.

6. Getting Started — Practical Steps

Recommended onboarding path for Varun, starting from zero Claude Code experience:

1
Start Claude Code on a familiar codebase. Install Claude Code via npm install -g @anthropic-ai/claude-code. Open your VM2-P-Taskers repo locally and run claude in that directory. Ask it to explain the codebase structure — this tests context loading and sets baseline expectations before any agentic tasks.
2
Write a CLAUDE.md for the VM2 repo. This is the most important setup step. Document the VM2 architecture, Railway deployment patterns, Changeis coding conventions, file naming standards, and GitHub commit message format. Every Claude session (and every teammate) will load this context automatically. A good CLAUDE.md is the difference between effective and frustrating agentic sessions.
3
Run a single-agent task first. Before using Agent Teams, get comfortable with single-session Claude Code on a real task: add a new field to the SAM.gov cron output, add tests, commit. This builds intuition for how Claude Code works before adding multi-agent complexity.
4
Enable Agent Teams. Add "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" to ~/.claude/settings.json. Install tmux (brew install tmux on Mac) for split-pane visibility — this makes observing multiple agents much easier than the in-process mode. Verify version: claude --version (must be ≥ 2.1.32).
5
First Agent Teams task: VM2 test coverage. This is the ideal first real use case — low risk, clear parallelism, immediate value. Prompt: "Create an agent team with 3 teammates. Teammate 1: write unit tests for the SAM.gov monitor cron. Teammate 2: write unit tests for the APFS monitor. Teammate 3: write integration tests for the portal.html generation pipeline. Use pytest. Coordinate if you encounter shared utility functions." Budget ~$5–8 for this task on Pro plan.
6
Upgrade plan if hitting limits. Pro plan supports 2–3 Agent Teams tasks per day. If you're doing active development work with Agent Teams, Claude Max 5× ($100/month) gives 8–10 complex tasks in a 5-hour window — more appropriate for sustained development sprints. Compare this against API direct access if you're doing very heavy use.
Recommended first week: Day 1–2: Install Claude Code, write CLAUDE.md, run single-agent tasks. Day 3–4: Enable Agent Teams, run the test coverage task. Day 5: Evaluate whether the token cost and workflow fit your development cadence. Decide on Pro vs. Max upgrade.
7. Recommendations for Varun
Do This Now

High Priority Actions

  • Install Claude Code — it complements VM2, doesn't replace it. Start with single-agent mode.
  • Write a CLAUDE.md for VM2-P-Taskers — document architecture, conventions, deployment patterns. This is foundational for any Claude Code work.
  • First Agent Teams use: test coverage — parallel test writing across VM2 cron modules. Low risk, clear value, good learning task.
  • Stay on Pro ($20/month) to evaluate fit before considering Max.
Do Not Do

Avoid These Mistakes

  • Don't use Agent Teams for proposals. Perplexity Computer is far better suited. Agent Teams is a coding tool.
  • Don't try to integrate Agent Teams into VM2 crons — it requires interactive terminal sessions and is not automatable today.
  • Don't skip writing CLAUDE.md. Without persistent project context, every session starts blind and agents will make incorrect assumptions about your architecture.
  • Don't run Agent Teams on sequential tasks — if tasks must happen in order, a single session is cheaper and more reliable.
The Right Mental Model

VM2 = your AI operating system — handles research, proposal work, opportunity monitoring, automated daily briefs, and all non-code tasks. It runs autonomously, 24/7, without your presence.

Claude Code Agent Teams = your AI development team — handles coding tasks interactively when you're doing development work. You're present, you can redirect agents, and you review the output. It's not automated; it's accelerated.

These tools are complementary, not competitive. The future vision — where VM2 can spawn a Claude Code Agent Team to implement a change autonomously and push to GitHub — is plausible in 12–18 months as Anthropic matures the API. For now, treat them as separate layers of your AI stack.
