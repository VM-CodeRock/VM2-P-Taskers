APFS Title & Notice Type Fix

Root cause analysis, code fix specification, and Notice Type field addition plan for the Changeis Daily BD Brief cron.

Bug: Field Name Mismatch Files: run_apfs_monitor_v3.py · generate_daily_brief.py Fix: 2 line changes

Executive Summary

Root Cause Confirmed

All 8 APFS opportunities in the March 25 (and March 24) daily brief show (Untitled) because the APFS monitor script reads r.get("title"), but the APFS JSON data actually stores the opportunity name under the key requirements_title. This mismatch causes the title field to always resolve to an empty string, which the brief generator then falls back to displaying as (Untitled).

Two files need updating: run_apfs_monitor_v3.py (the monitor/data ingestion script) and optionally generate_daily_brief.py (the HTML renderer). A third enhancement — adding Type of Notice to each opportunity card — requires changes to the same two files plus the HTML render function.

Root Cause Analysis

1. APFS Data Structure

The DHS Acquisition Planning Forecast System (APFS) table column is named "Requirements Title". When this data is fetched and stored as JSON (e.g., in apfs_current_raw.json), the key becomes requirements_title in snake_case, consistent with all other APFS fields (requirements_contact_first_name, requirements_office, etc.).

APFS Website ColumnJSON Field Name (correct)What monitor uses (wrong)
Requirements Titlerequirements_titletitle ❌
Componentorganizationorganization ✅
NAICSnaicsnaics ✅
Dollar Rangedollar_rangedollar_range ✅
Contract Statuscontract_statusNot fetched ❌
Contract Typecontract_typecontract_type ✅
Estimated Solicitation Releaseestimated_release_dateest_solicitation_release ❌

2. The Exact Bug

In backups/scripts/run_apfs_monitor_v3.py, line 161:

    for change_type, recs in [("NEW", new_recs), ("UPDATED", updated_recs)]:
        for r in recs:
-           title = r.get("title") or ""             # ❌ WRONG — field doesn't exist in APFS JSON
+           title = r.get("requirements_title") or "" # ✅ CORRECT — matches actual APFS JSON key
            req = r.get("requirement_description") or ""
            query = f"{title}. {req}".strip()

And in the opportunities_out dict at line 191, the title is saved as:

-   "title": title,  # Always "" because r.get("title") returns None
+   "title": title,  # Now correctly populated from requirements_title

The generate_daily_brief.py then calls opp.get("title", "(Untitled)"), which falls back to "(Untitled)" because the title was never stored.

3. Secondary Field: est_solicitation_release

The monitor stores r.get("est_solicitation_release") but the actual APFS JSON key is estimated_release_date. This causes the solicitation release date to also be blank in the brief. This should be fixed in the same pass.

4. Confirmation via Reference Script

The older backups/scripts/build_opportunity_db.py correctly uses rec.get("requirements_title", "") (line 222) and rec.get("estimated_release_date", "") (line 230). The monitor was written separately and used the wrong key names.

Notice Type Field — APFS vs SAM.gov

APFS and SAM.gov use different terminology for what is conceptually "notice type."

SourceField NameExample ValuesMeaning
APFScontract_statusNew Requirement, Re-competitionIs this a brand-new contract or a recompete of an existing one?
APFScontract_typeFirm Fixed Price, T&M, IDIQContract pricing mechanism
SAM.govnotice_typeSources Sought, Solicitation, Pre-Solicitation, Award NoticeStage in the procurement process

For APFS opportunities, display Contract Status (New Requirement vs Re-competition) since APFS data is pre-solicitation forecast data — the solicitation type (RFP, Sources Sought) is not yet determined. For SAM.gov, display Notice Type which is the formal procurement notice category.
