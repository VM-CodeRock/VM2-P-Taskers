1 · What Is Hermes 4?

Hermes 4 is actually two distinct but related products from Nous Research. Understanding the distinction is critical before deciding how to use it.

Product A

Hermes 4 Models

A family of open-weight hybrid reasoning LLMs (70B and 405B parameters) based on Llama 3.1. These are language models — not agents. Available via the Nous Portal API and OpenRouter.

Think of it as: A Claude/GPT alternative you can call via API — steerable, uncensored, strong at reasoning and structured output.
Product B

Hermes Agent

An open-source autonomous AI agent framework built around the Hermes 4 models. Self-hosted on a server/VPS. Persistent memory, skill creation, multi-platform messaging, and sub-agent spawning. GitHub (17.7k ★)

Think of it as: A self-hosted version of Perplexity Computer's agent loop — but you run the infrastructure.
Bottom line for Varun: The website hermes4.nousresearch.com covers both. The Todoist question is primarily about Hermes Agent (the agent framework) — which uses Hermes 4 models as its brain, but can also use Claude Opus 4.6, GPT-5.4, or any of 200+ models via OpenRouter.

Key differentiator from other agents: Hermes Agent has a closed learning loop — it autonomously creates skills from completed tasks, self-improves those skills during use, and maintains full cross-session memory via FTS5 search with LLM summarization. It's designed to compound in capability over time.

Recommended multi-model pattern for Varun's workflow: Orchestrate with Opus 4.6 for strategy and decisions → delegate sub-tasks to Hermes-4-70B for speed-sensitive lookups → use Hermes-4-405B for deep analysis and structured proposal content. This mirrors exactly how VM2 works today with sub-agents.

Sub-agent capacity: Current Hermes Agent supports up to 3 concurrent sub-agents in batch mode (delegate_task). A GitHub issue (filed March 2026) is tracking expansion to 5–30 workers for parallel code migration and larger orchestration trees. Sub-agents are fully isolated — the parent never sees a child's full token stream, preventing context poisoning.

Sub-Agent Architecture

How Isolation Works

  • Parent spawns isolated child processes
  • Each child has its own context window
  • No context bleeding between agents
  • Parent orchestrates results only
  • Enables token-heavy parallel work
Current limit: 3 concurrent (expanding to 5–30 in roadmap)
Memory Architecture

Two-Tier Memory

  • Tier 1: Local Markdown files (fast inject, ~800 tokens)
  • Tier 2: SQL (FTS5) for deep cross-session recall
  • Auto-compression at 85% context fill
  • Honcho user modeling across sessions
  • Context never lost between restarts
Unlike: Stateless ChatGPT/Claude sessions
Model Switching

Zero Lock-In

  • Switch with hermes model
  • Per-task model override supported
  • OpenAI-compatible client interface
  • 300+ models via OpenRouter
  • Also: your own local endpoint
No code changes required to switch models
4 · Roadmap & Implementation Plan (with Mission Control)

Below is a phased plan to evaluate, deploy, and integrate Hermes Agent alongside your existing VM2 / Perplexity Computer setup. This is additive — not a replacement.

Phase 1 · Week 1 — Evaluate the Models

Try Hermes 4 API Without Committing to Infrastructure

  • Create an account at portal.nousresearch.com
  • Test Hermes-4-405B in Nous Chat (chat.nousresearch.com) — free trial available
  • Send 3–5 GovCon-style prompts: proposal narrative, SAM.gov opportunity analysis, IT strategy memo
  • Compare output quality vs. Claude Opus 4.6 on the same prompts
  • Key question: Is Hermes-4-405B at $0.37/1M output good enough to replace Opus for non-orchestration tasks?
Phase 2 · Week 2 — Pilot the Agent on a $5 VPS

Stand Up Hermes Agent — Isolated from Production

  • Spin up a DigitalOcean/Hetzner VPS ($5–$10/mo). Do NOT use your primary machine.
  • Install: curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
  • Configure with Telegram as the primary interface (simplest gateway to test)
  • Set your model to Hermes-4-70B for the pilot (low cost, fast feedback)
  • Enable Docker backend for isolation: hermes config set terminal.backend docker
  • Run hermes setup and configure only your Telegram user ID in allowed users
  • Test: ask it to research a SAM.gov opportunity, summarize a PDF, draft a paragraph
Phase 3 · Week 3 — Gmail + Skills Integration

Connect Email and Build First Changeis Skills

  • Connect Gmail via IMAP/SMTP with an App Password (see Section 6)
  • Set up email gateway: have Hermes watch a dedicated inbox (e.g., hermes@changeis.com or alias)
  • Create your first custom skill: GovCon Opportunity Brief — takes a SAM.gov URL, returns a formatted assessment
  • Create second skill: Daily Changeis Brief — mirrors what VM2 already does, but runs on Hermes infrastructure
  • Test sub-agent delegation: have the orchestrator spawn two parallel sub-agents for different aspects of an opportunity analysis
Phase 4 · Week 4 — Mission Control Integration

VM2 + Hermes Agent as Complementary Mission Control

  • VM2 (Perplexity Computer) = Research, document creation, GitHub/Dropbox workflows, portal deliverables, Todoist management
  • Hermes Agent = 24/7 persistent monitoring, email routing, Telegram access while mobile, scheduled automations, local data privacy
  • Integration point: Hermes Agent can trigger VM2 tasks via email (send to vm2.viru@gmail.com with "VM2" in subject)
  • Integration point: VM2 can reference Hermes Agent outputs saved to shared storage
  • Upgrade orchestrator to Claude Opus 4.6 for production use (hermes model → OpenRouter → anthropic/claude-opus-4.6)
  • Deploy cron schedules: daily GovCon brief at 7 AM, nightly email digest, weekly backups
Phase 5 · Month 2 — Scale & Optimize

Full Production Stack

  • Migrate to Modal (serverless) for zero idle cost — agent hibernates, wakes on demand
  • Add Slack gateway for team-facing interactions at Changeis
  • Build 3–5 additional GovCon skills: proposal section drafting, NAICS code analysis, past performance builder
  • Evaluate multi-agent architecture (Phase 4 roadmap on GitHub) as it ships — targeting 5–30 parallel sub-agents
  • Assess whether to sunset any VM2 workflows that Hermes handles better at lower cost
5 · Is It Safe to Use?

Hermes Agent has a five-layer defense-in-depth security model. For a government IT consulting context, here's the honest assessment:

Security LayerWhat It DoesVerdict for GovCon
User AuthorizationAllowlists + DM pairing with cryptographic codes (1-hr TTL, rate limited, audit logged)Strong — only your user ID can interact
Dangerous Command ApprovalHuman-in-the-loop approval for destructive shell ops (rm -rf, DROP TABLE, etc.) with always/once/session modesStrong — you stay in control of risky actions
Container Isolation (Docker)Drops all Linux capabilities, no-new-privileges, PID limit 256, tmpfs only, read-only filesystemStrong — production Docker mode is hardened
MCP Credential FilteringStrips API keys, tokens, secrets from sub-process environments. Credential files mounted read-only.Good — keys don't leak to child processes
Prompt Injection ProtectionTirith pre-exec scanner + context file scanning for homograph attacks, pipe-to-shell, invisible UnicodeAdequate — covers common vectors
Data ResidencyRuns on your own server. Your data never goes to Nous Research servers (only model API calls go to provider).Strong — critical for CUI/sensitive GovCon data
SSRF ProtectionBlocks access to RFC 1918 private nets, cloud metadata endpoints (AWS IMDSv1/v2, GCP, etc.)Good — prevents agent from pivoting to internal network
GovCon-specific caution: If you handle CUI (Controlled Unclassified Information) or data subject to FedRAMP/FISMA requirements, you must ensure the model API provider (Nous Portal / OpenRouter / Anthropic) meets your data handling obligations. The agent framework itself is local and safe; the model API calls send your prompt content to external servers. For sensitive work, use a locally hosted model (Hermes-4-405B weights are available on Hugging Face) instead of the cloud API.

Bottom line: For day-to-day GovCon consulting tasks (opportunity research, proposal drafting, briefings, scheduling), Hermes Agent is reasonably safe when deployed in Docker mode with explicit user allowlists. It is not pre-certified for any federal security framework. Treat it the same as any other cloud AI tool — don't feed it classified or CUI data through the cloud API.

6 · Can You Give It Access to Your Email?

Yes — email is a first-class, natively supported gateway in Hermes Agent. Here's exactly how it works and how to set it up safely:

How it works

IMAP/SMTP Gateway

  • Hermes polls your inbox via IMAP (configurable interval)
  • Matches incoming emails to skills and automations
  • Can send replies, summaries, or triggered actions via SMTP
  • Supports Gmail (App Password), Outlook, any standard mail server
  • Cross-platform continuity — same conversation thread from email or Telegram
Setup for Gmail

Step-by-Step

  • Go to Google Account → Security → 2-Step Verification → App Passwords
  • Generate an App Password for "Mail"
  • Run hermes gateway setup and choose Email
  • Enter: IMAP host imap.gmail.com, port 993
  • SMTP host smtp.gmail.com, port 587
  • Use App Password (NOT your Google password)
  • Restrict to specific sender allowlist for security
Recommended approach for Varun: Create a dedicated Gmail alias (e.g., hermes.changeis@gmail.com) rather than granting access to your primary varun@changeis.com. Route email briefs, opportunity alerts, and client messages to the alias. This keeps production email separate and limits blast radius if something goes wrong.

The existing VM2 pipeline already uses vm2.viru@gmail.com for the email task intake channel. Hermes Agent can complement this — watching a different inbox for real-time response and continuous monitoring, while VM2 handles deep research and document deliverables.
