VM2 Thread Tag Mechanism

Design specification for persistent, cross-deliverable analysis threads — Gmail, Todoist, and portal integration
Date: May 5, 2026
Pilot Thread: FAA APEX (faa-apex)
Status: Implemented — v1
Task Name: VM2 Thread Tag Mechanism - APEX Pilot

1. Overview

Varun's VM2 workflow produces many deliverables over time. Some of those deliverables are logically part of the same ongoing analysis — for example, three separate briefs all tracking the FAA APEX opportunity. Without a grouping mechanism, these deliverables are connected only by text search, not by explicit structure.

The Thread Tag system adds a lightweight, additive dimension — a thread slug — that binds related deliverables together without changing how any existing file or workflow operates. It surfaces as:

  • A data-thread="<slug>" attribute on portal.html table rows
  • A "thread": "<slug>" field inside each deliverable's context slug JSON
  • A [thread:X] marker in Gmail subject lines (auto-extracted by intake cron)
  • A vm2-thread-X label in Todoist (or inline in task description)
  • A dedicated thread aggregator page (e.g., faa-apex-thread.html) that renders all tagged deliverables in a chronological timeline

Design principle: Threads and versions are orthogonal. Versions chain a single deliverable (v1 → v2 → v3). Threads aggregate multiple different deliverables. A deliverable can have both a version chain and a thread tag simultaneously.

2. Data Model

2.1 Thread Slug Convention

Slugs are kebab-case, lowercase, descriptive. Examples:

2.2 portal.html Row Attribute

Add data-thread="<slug>" to the <tr> element, alongside existing data-date, data-type, and data-updated:

<tr data-date="2026-05-04" data-type="Research Brief"
    data-updated="2026-05-04" data-thread="faa-apex">

    FAA APEX Industry Day — Analysis &amp; Next Steps</a></td>
  ...
</tr>

Add a "thread" field to the vm2-context-data script block, adjacent to related_deliverables:

  ...

3. Gmail Workflow

3.1 Subject Line Convention

To tag a new Gmail-submitted task to a thread, include [thread:slug] anywhere in the subject line:

Subject: VM2 - [thread:faa-apex] Post-event debrief APEX Industry Day
Subject: VM2 - [thread:faa-apex] Teaming shortlist follow-up
Subject: VM2 - [thread:apex-framework] New use cases for agentic org model

The intake cron (cron job 2da8354a) extracts the tag with a single regex and injects it into:

  • The data-thread attribute on the new portal row
  • The "thread" field in the new deliverable's context slug

3.2 Intake Cron Extraction Logic

# Pseudocode — intake cron 2da8354a
subject = email.subject  # e.g. "VM2 - [thread:faa-apex] Post-event debrief"

thread_match = re.search(r'\[thread:([a-z0-9\-]+)\]', subject, re.IGNORECASE)
thread_slug = thread_match.group(1).lower() if thread_match else None

# Inject into task context
task_context['thread'] = thread_slug  # e.g. "faa-apex"

# Inject into portal row (when generating the new <tr>)
thread_attr = f'data-thread="{thread_slug}"' if thread_slug else ''
row_html = f'<tr data-date="{date}" data-type="{type_}" data-updated="{date}" {thread_attr}>'

3.3 Reply-to-Thread Convention

When replying to an existing VM2 email thread (e.g., replying to the APEX Industry Day email), the thread tag is automatically inherited if the original email contained [thread:X] in the subject. No additional marker is needed in the reply body — the intake cron can detect the thread ID from the Gmail thread context.

Alternative: include [thread:faa-apex] anywhere in the reply body's first paragraph as a fallback extraction point.

4. Todoist Workflow

4.1 Label Convention (Option A — Recommended)

Create a single Todoist label vm2-thread. When creating a VM2 task that belongs to a thread, apply this label and put the thread slug as the first line of the task description:

Task name:  FAA APEX — post-event debrief
Labels:     vm2, vm2-thread
Description (first line):  thread:faa-apex
            (rest):         Write up key takeaways from Industry Day May 15.
                            Include teaming status + acquisition vehicle update.

The intake cron parses the task description's first line for thread:<slug> and injects it accordingly.

4.2 Per-Thread Label Convention (Option B)

Create individual labels per thread: vm2-thread-faa-apex, vm2-thread-apex-framework, etc. Apply the relevant label to each task. The intake cron strips vm2-thread- prefix to extract the slug. Simpler extraction but creates label sprawl over time.

Recommendation: Use Option A (single vm2-thread label + description-first-line slug) to minimize Todoist label proliferation.

4.3 Todoist Extraction Pseudocode

# When processing a Todoist task with label "vm2-thread"
description_lines = task.description.strip().split('\n')
first_line = description_lines[0].strip()  # e.g. "thread:faa-apex"

thread_slug = None
if first_line.startswith('thread:'):
    thread_slug = first_line.split('thread:')[1].strip().lower()

task_context['thread'] = thread_slug

5. Portal Changes (portal.html)

5.1 Thread Filter Dropdown

Add a Thread dropdown filter to the filter-bar section, above or below the existing type pills. The dropdown dynamically reads all distinct data-thread values from table rows and offers them as filter options:

<!-- Add to filter-bar, after search-row -->
<div class="thread-row">
  <label for="threadFilter">Thread:</label>
  <select id="threadFilter">
    <option value="">All threads</option>
    <!-- populated by JS from data-thread attribute values -->
  </select>
  <a id="threadViewLink" href="#" style="display:none">View thread page ↗</a>
</div>

<script>
// Collect distinct thread slugs
const threads = new Set();
document.querySelectorAll('tr[data-thread]').forEach(r => threads.add(r.dataset.thread));
const sel = document.getElementById('threadFilter');
[...threads].sort().forEach(t => {
  const opt = document.createElement('option');
  opt.value = t; opt.textContent = t;
  sel.appendChild(opt);
});
// Filter on change
sel.addEventListener('change', () => {
  const slug = sel.value;
  document.getElementById('threadViewLink').style.display = slug ? 'inline' : 'none';
  document.getElementById('threadViewLink').href = slug + '-thread.html';
  applyFilters(); // extend existing applyFilters to check data-thread
});
</script>

5.2 applyFilters Extension

Extend the existing applyFilters() function with a single additional condition:

// Add inside applyFilters(), after the existing date/type checks:
const threadFilter = document.getElementById('threadFilter')?.value;
if (threadFilter && row.getAttribute('data-thread') !== threadFilter) visible = false;

6. Thread View Page

6.1 Page Pattern

Each active thread gets a dedicated HTML page named <slug>-thread.html (e.g., faa-apex-thread.html). The page:

  • Renders a header card summarizing the thread topic (opportunity status, key dates, context)
  • Reads the portal.html table client-side via fetch() and filters rows where data-thread="faa-apex"
  • Renders a chronological timeline of all matching deliverables
  • Links to the individual deliverable for each entry
  • Refreshes automatically when portal.html is updated — no separate maintenance

6.2 Example Markup (Abbreviated)

<!-- faa-apex-thread.html (simplified skeleton) -->
<div class="thread-header">
  <h1>FAA APEX — Thread View</h1>
  <p>NavTac II TO-16 successor · $5–20M estimate · Industry Day May 15</p>
</div>

<div id="timeline">Loading...</div>

<script>
fetch('/portal.html')
  .then(r => r.text())
  .then(html => {
    const doc = new DOMParser().parseFromString(html, 'text/html');
    const rows = [...doc.querySelectorAll('tr[data-thread="faa-apex"]')];
    rows.sort((a,b) => b.dataset.date.localeCompare(a.dataset.date));
    const tl = document.getElementById('timeline');
    tl.innerHTML = rows.map(row => {
      const a = row.querySelector('a');
      return `<div class="tl-item">
        <div class="tl-date">${row.dataset.date}</div>
        <div class="tl-title"><a href="${a.href}">${a.textContent}</a></div>
      </div>`;
    }).join('');
  });
</script>

7. Versioning (Orthogonal to Threads)

Threads and versions serve different purposes and stack independently:

DimensionMechanismPurposeExample
VersionFilename suffix -v2.html; prior_versions chain in context slugTrack revisions of a single deliverablefaa-apex-industry-day-v2.html
Threaddata-thread attribute; "thread" in context slugGroup multiple different deliverables on one topicBoth v1 and v2 have data-thread="faa-apex"

A versioned deliverable retains its thread tag across all versions. The thread page shows all versions as distinct timeline entries (since they are distinct deliverable files), with their dates in chronological order.

8. Migration Path for Existing Deliverables

Backfilling thread tags is additive and non-breaking. Steps:

  1. Identify clusters: Group existing deliverables by topic (APEX, govcon, daily briefs, internal frameworks, etc.)
  2. Assign slugs: Pick a stable kebab-case slug per cluster
  3. Edit portal.html rows: Add data-thread="<slug>" to each relevant <tr>
  4. Edit context slugs (optional): Add "thread": "<slug>" to each deliverable's vm2-context-data JSON block
  5. Create thread pages: Generate <slug>-thread.html for each cluster with a relevant header card

Deliverables without a thread tag are unaffected — the data-thread attribute is optional and the portal filter shows all rows by default.
