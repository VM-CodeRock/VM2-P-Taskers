Golf Rounds Tracker

Mobile-first PWA for hole-by-hole scoring zone analysis and round notes
May 10, 2026
Created: 2026-05-10
Executive Summary

A self-hosted, mobile-first web application to capture per-round and per-hole golf data on the course, edit everything later, and run reports on demand. The deliverable comprises three things: (1) this PRD with full architecture, (2) a working v0 prototype already deployed and usable from a phone today, and (3) an upgrade path to a Railway-hosted FastAPI + Postgres service when persistence across devices is needed.

v0 (shipped today): Static SPA backed by client-side storage. Captures everything in the spec, auto-saves between holes, supports Yes/No toggles, putt counts, hole-paginated entry, full-round reports, and cross-round stats with course/date filters. Editable end-to-end. Deployable to Railway as a static site (or already live on the included preview link).

v1 (upgrade path): Same UI, swap the storage layer for a FastAPI backend with SQLite (file in Railway volume) or Postgres (Railway-managed). REST API with bearer token auth (single-user). Multi-device sync and richer custom reports.

Why this shape. The spec asked for an app that works on a phone, captures structured data hole-by-hole, lets the user edit anything later, and reports on demand. A static SPA with client-side storage satisfies all functional requirements with zero backend cost; the FastAPI upgrade is a one-evening lift when cross-device sync becomes valuable.

Product Requirements Document

1. Goals

  • Primary: Capture hole-by-hole scoring-zone data and notes during a live round on a phone, with one hole visible at a time and one tap per Yes/No.
  • Editable: Every value (round and hole) is editable post-round; no destructive flows.
  • Insightful: Compute per-round counts (SZ Yes/No, DSZ Yes/No, total putts, putts-close Yes/No) automatically and surface them on the round report.
  • Aggregable: Run custom reports across rounds (filter by course, date range) on demand.
  • Deployable: Ship via the existing Git + Railway pipeline; no new infrastructure.

2. Users

  • Primary user: Varun (single user). One round at a time, played on a smartphone.
  • Devices: iOS Safari (primary) and Android Chrome. Optional desktop access for editing/reports.
  • Network: Often spotty mid-round. App must work offline once loaded; no blocking network calls during scoring.

4. Non-Functional Requirements

  • Mobile-first: Single-column layout, 44px+ tap targets, large stepper buttons, sticky topbar, no horizontal scroll.
  • Offline: All scoring works without network. v0 uses a service worker to cache the shell; data is local. v1 will queue writes when offline.
  • Persistence: v0 — client-side store, never expires. v1 — Postgres or SQLite-on-volume, daily logical backups via Railway cron.
  • Editable forever: No fields ever become read-only. No computed-stat persistence — recomputed from raw data on every render.
  • Latency: Hole transitions < 100 ms. App shell loads < 1 s on 4G.
  • Accessibility: Visible focus, aria-pressed on Yes/No toggles, semantic labels, color-independent state (text + color).
  • Privacy: Single-user app; v1 protected by bearer token + Railway basic auth.
  • Cost: v0 = $0 (static hosting). v1 = within existing Railway plan.
Data Model

Three logical entities. Computed stats are not stored — they are derived from raw hole rows on read so any edit immediately reflects in reports.

// round
{
  "id":         "r_xxxxx",        // UUID / nanoid
  "date":       "2026-05-10",     // ISO date, default today
  "course":     "Westfields GC",  // free text
  "tees":       "Blue",           // free text
  "notes":      "windy, partner: ...",
  "holes":      [Hole x18],       // one per hole 1..18
  "created_at": "2026-05-10T14:00:00Z",
  "updated_at": "2026-05-10T18:33:00Z"
}

// hole (always 18 per round)
{
  "hole":         1,              // 1..18
  "entered_sz":   true | false | null,
  "down_sz":      true | false | null,
  "putts":        0..15,
  "putts_close":  true | false | null,
  "notes":        "missed long",
  "filled":       true            // any field set OR notes non-empty
}

// computed_stats (derived, never stored)
{
  "filled":         12,
  "sz_yes":         8,  "sz_no":  4,  "sz_pct":  67,
  "dsz_yes":        5,  "dsz_no": 3,  "dsz_pct": 63,   // % of SZ-entered holes
  "putts_total":    29, "avg_putts": "2.42",
  "close_yes":      7,  "close_no": 3, "close_pct": 70  // % of holes with putts > 0
}

Why nullable Yes/No. A hole the user hasn't reached yet must be distinguishable from a deliberate "No". Three states (yes / no / unset) keep stats honest.

API Design (v1 — FastAPI)

v0 ships without a backend. v1 introduces a thin REST API. All endpoints require Authorization: Bearer <token>.

MethodPathPurpose
GET/api/roundsList all rounds (sorted date desc). Supports ?course=&from=&to=.
POST/api/roundsCreate round. Body: { date, course, tees, notes }. Server creates 18 empty holes.
GET/api/rounds/{id}Fetch full round with holes + computed stats.
PATCH/api/rounds/{id}Update round metadata (date, course, tees, notes).
DELETE/api/rounds/{id}Delete round (cascades to holes).
PATCH/api/rounds/{id}/holes/{n}Update one hole. Body: any subset of { entered_sz, down_sz, putts, putts_close, notes }.
GET/api/statsAggregate stats. Query: ?course=&from=&to=. Returns rolled-up SZ%, DSZ%, putts, close%.
GET/api/exportFull JSON dump (for backup).
POST/api/importBulk import (e.g. migrating v0 client data to v1).

All endpoints return JSON. Errors use standard HTTP status with { "detail": "…" } body (FastAPI default).

Tech Stack Recommendation

Frontend

  • v0 (shipped): Single-file vanilla HTML/CSS/JS — no build, no framework, no node toolchain. ~38 KB total. Service worker for offline shell. PWA manifest for install-to-home-screen.
  • v1: React + Vite (mobile-first PWA). TypeScript. Tailwind for utilities. vite-plugin-pwa for service worker + manifest. Same UI/UX as v0; only the data layer changes.

Backend (v1)

  • FastAPI (Python 3.11) — async, fast, batteries-included. SQLAlchemy 2.0 + Alembic migrations.
  • Database: SQLite on a Railway volume for simplicity, or Railway-managed Postgres if you want more headroom. Both work; SQLite is fine for < 10k rounds.
  • Auth: Single static bearer token in env var. No user table; Railway basic auth is the outer fence.
  • Server: uvicorn behind Railway's proxy. Static SPA served from /static by FastAPI.

Hosting

  • Railway — already set up. Same project, new service or deployed alongside vm2-p-taskers.
  • v0 deploy: Static folder; Railway serves index.html. Or piggyback inside vm2-p-taskers at /golf/.
  • v1 deploy: Procfile or Railway's auto-detect for FastAPI. Env vars for DATABASE_URL and API_TOKEN.

Spec-Driven Engineering Plan

Each phase has explicit acceptance criteria. No phase ships unless all its checks pass on a real phone.

Phase 0 — Spec freeze (today)

  • This document is the spec. Lock the data model, the screen list, and the per-hole field set.
  • Acceptance: data model + REST contract + screen wireframes all reviewed and signed off.

Phase 1 — v0 static SPA (today, shipped)

  • Single-file app with all five screens, persistent client storage, manifest + service worker.
  • Acceptance: Round CRUD + 18-hole capture + report + stats dashboard, all on a phone, all editable.
  • Status: shipped

Phase 2 — Polish + offline UX

  • Course autocomplete from history; tee suggestions; haptic feedback on Y/N tap; landscape lock.
  • "Resume" prompt if an unfinished round exists.
  • Acceptance: round started in airplane mode, resumed online.

Phase 3 — Backend (FastAPI + DB)

  • Stand up FastAPI service on Railway. Implement the API contract above. SQLAlchemy models + Alembic init migration.
  • Bearer-token auth via env var.
  • Acceptance: curl CRUD round → returns expected JSON; OpenAPI schema matches this PRD.

Phase 4 — Sync (frontend ↔ backend)

  • Frontend gains a "data adapter" interface; swap client store for HTTP client.
  • One-time migration: import client JSON via POST /api/import.
  • Optimistic writes with offline queue; replay on reconnect.
  • Acceptance: Edit on phone, refresh on laptop, see same data.

Phase 5 — Custom reports

  • Saved report views (e.g. "Last 10 rounds at Westfields"); compare two date windows; trend sparkline.
  • Acceptance: User can build, save, and reload a parameterised report.

Build & deploy flow

  1. git push origin main → Railway auto-builds.
  2. Build: cd frontend && npm run build; pip install -r backend/requirements.txt.
  3. Run: uvicorn app.main:app --host 0.0.0.0 --port $PORT. FastAPI serves SPA at / and API at /api/*.
  4. v0 alternative: drop the golf-tracker/ static bundle into the existing vm2-p-taskers repo at /golf/ and ship today.
Acceptance Criteria
  • I can open the app on my phone, tap +, and start a new round in under 5 seconds.
  • Round date defaults to today; I can override with a calendar picker.
  • One hole is visible at a time; I can advance with a single tap; data is saved between holes.
  • Yes/No toggles use large tap targets and indicate state with both color and text/aria-pressed.
  • Putt count increments with thumb-reachable +/−; values 0–15 supported.
  • I can open a finished round and see SZ Y/N counts, DSZ Y/N counts, total putts, and close-putt counts — without doing math.
  • I can edit any field of any round at any time; computed stats update immediately.
  • I can run a custom report across all rounds, filter by course and date range, and see aggregate SZ%, DSZ%, avg putts, close%.
  • I can export my data as JSON for backup at any time.
  • The app works without network connectivity for an entire round once loaded.
  • Everything is deployed via the existing Git → Railway pipeline; no new infra required.
