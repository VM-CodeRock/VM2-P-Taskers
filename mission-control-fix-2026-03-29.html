Mission Control — Cron State Fix

Why 0/3 cron state files were loading — root cause & fix
March 29, 2026 · vm2.viru@gmail.com
Root Cause
The cron IDs in _system/mc-config.json did not match any actual folder names in /cron_tracking/.

The Mission Control dashboard fetches cron state from paths like /cron_tracking/{id}/state.json on the Railway server. The config file had placeholder IDs (6a1f1c3b, d56ab3fd, aa619acf) that were never created as actual directories. The fetch returned 404 for all three, causing the "0/3 loaded" display.

There was also a secondary issue: the actual cron state files live at /home/user/workspace/cron_tracking/{name}/ in the Perplexity Computer workspace, but Railway only serves files that are inside the VM2-P-Taskers Git repo. The state files had never been copied into that repo.

Long-Term Fix — Auto-Sync State Files

The cron state files in /home/user/workspace/cron_tracking/ are updated by each cron run but are NOT automatically pushed to GitHub. This means the Mission Control view will go stale until the next GitHub push.

Recommended solution: Each cron job (Gmail intake, Todoist VM2, MATS/MOTUS) should copy its updated state.json into the corresponding VM2-P-Taskers/cron_tracking/{name}/state.json and push it to GitHub as part of its normal wrap-up. This is a one-line addition to each cron's existing GitHub push step.

This update has been noted in the Gmail intake cron's state file. The next cron sessions should implement this sync step.
