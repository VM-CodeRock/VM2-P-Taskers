01 System Overview & Architecture

What is VM2?

VM2 ("Varun Malhotra 2") is an automated Chief-of-Staff AI system built on Perplexity Computer for Varun Malhotra, President of Changeis Inc. It runs 14 scheduled tasks that automate:

  • Daily Executive Agenda — Morning briefing with calendar, email triage, and task priorities
  • Federal Opportunity Intelligence — Monitoring DHS APFS and SAM.gov for contract opportunities, RAG-scoring them against Changeis past performance, and generating deep-dive analyses
  • Task Automation — Executing tasks from Todoist (VM2 label) and Gmail (vm2.viru@gmail.com)
  • Compliance & Governance — Nightly audits, weekly digests, monthly integrity checks, deliverable integrity audits
  • Business Development — YPO introduction tracking, Todoist auto-helper suggestions, MAPS amendment monitoring

DATA COLLECTION (run in parallel):
1. OUTLOOK CALENDAR: Use search_calendar (source: outlook) for today's events

   - One with label "VM2" to get agent tasks
   - One with no filters to get all tasks (then client-side filter for today's due/overdue items)
   NOTE: The Todoist connector does NOT support due-date filters. You must fetch broadly and filter in code: parse each task's due.date field, keep only tasks where due.date <= today OR due.date is null (no due date but not completed). Also check for @VEB label.

SYNTHESIS: De-duplicate across calendar + email + Todoist. Write the Commanding Theme. Enrich meetings with badges, attendees, context.

OUTPUT GENERATION:

- ALSO save the HTML to /home/user/workspace/VM2-P-Taskers/daily-executive-agenda-YYYY-MM-DD.html

EMAIL DELIVERY:

  1. Save the HTML brief as an attachment file

Design Principle: Threads vs. Versions

Threads and versions are orthogonal. Threads aggregate different deliverables under a topic. Versions chain a single deliverable over time. A deliverable can belong to a thread and also have prior versions — they do not conflict. System docs have data-thread="" (no thread).

  10. Outlook plain-text body — Outlook connector only supports plain text. Always use attachment for HTML.
  11. Todoist connector limit — Max 50 tasks. Always use direct REST API with cursor pagination as primary method. Connector is fallback only.

Key Lessons Learned

  • Auto-dive was removed (v2.2) because it created tasks Varun didn't request. The daily brief is INFORMATIONAL ONLY.
  • The vm2-recurring label system prevents catch-up loops for recurring tasks.
  • Delivery logs (appended to Todoist task descriptions) provide a running history of recurring task outputs.
  • The Todoist proxy on Railway (/api/todoist/) keeps the API token out of page source.
  • portal.html must insert rows after <tbody id="tableBody"> with proper data attributes for filtering to work.
  • (v2.0) Gmail state dual-write: Both gmail_intake/state.json and 6a1f1c3b/state.json must be kept in sync.
  • (v2.0) Todoist ID migration: When a cron is recreated with a new ID, the old ID's state files must be migrated on first run.
  • (v2.0) No-confirmation policy eliminates the need for confirm_action calls in all automated crons.
  • (v2.0) MAPS amendment deliverables should be consolidated into one document per solicitation.
  • (v2.3) Thread Tag Mechanism is fully additive — data-thread="" on all existing rows (empty string = no thread). Portal filter reads only non-empty values. No existing JS breaks.
  • (v2.3) Auto-Helper REST cursor pagination: use https://api.todoist.com/api/v1/tasks with cursor for pages, not the connector.
  • (v2.3) Combined Daily Brief builder has a context-slug regression — manually backfill if slug fields are missing.
