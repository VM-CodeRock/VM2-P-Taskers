SharePoint CSV Consolidation Agent — Solution Architecture

1. Overview

This solution automates the consolidation of CSV files dropped into dated subfolders within a SharePoint document library. Each day, new CSV files are deposited into a subfolder named by date (e.g., 2026-03-18/). The agent detects new subfolders, waits 30 minutes after the first file appears to allow all uploads to complete, then downloads and concatenates all CSVs into a single normalized output file suitable for Tableau analysis.

Key constraints:
- No data transformation — pure consolidation (append rows, align columns)
- 30-minute grace period after first file detection in a new dated subfolder
- Output must be a single CSV with consistent columns across all source files
- Must handle schema variations (missing columns filled with blanks, extra columns preserved)

3. Component Breakdown

3a. SharePoint Monitoring — Detecting New Dated Subfolders and File Drops

The agent polls a configured SharePoint document library folder for subfolders matching a date pattern (e.g., YYYY-MM-DD). On each poll cycle:

  1. List subfolders in the root drop folder via the SharePoint/Graph API.
  2. Filter to folders matching the expected date naming convention.
  3. Compare against a local state file that tracks which folders have already been processed.
  4. Identify new folders — any dated folder not yet marked as "processed" or "pending."
  5. List files in each new folder to confirm at least one .csv file exists.

API calls used:
- GET /sites/{site-id}/drive/root:/{path}:/children (Microsoft Graph)
- Or /_api/web/GetFolderByServerRelativeUrl('{path}')/Folders (SharePoint REST)

3b. 30-Minute Wait Mechanism

When a new dated subfolder is first detected with files:

  1. Record the detection timestamp in the state file: {"2026-03-18": {"first_seen": "2026-03-18T08:05:00Z", "status": "pending"}}.
  2. On each subsequent poll, check if now - first_seen >= 30 minutes.
  3. Once the 30-minute threshold is met, proceed to consolidation.
  4. After successful consolidation, update status to "processed".

This design is resilient to agent restarts — the state file persists the timer. The agent does not need to stay running continuously; it can be invoked on a schedule (e.g., every 5 minutes) and will pick up where it left off.

3c. CSV File Consolidation Logic

  1. Download all .csv files from the dated subfolder to a local temp directory.
  2. Read each CSV with pandas.read_csv() using dtype=str to prevent any type coercion (no data transformation).
  3. Concatenate all DataFrames with pd.concat(dataframes, ignore_index=True, sort=False).
  4. sort=False preserves column order from the first file.
  5. Missing columns in any file are filled with empty strings (NaN → "").
  6. Add metadata columns (optional, configurable):
  7. _source_file: original filename for traceability.
  8. _source_folder: the dated folder name.
  9. Write the consolidated CSV to the output location.

3d. Normalized Output Format

The output is a single CSV file with:
- Union of all columns across all input files (superset schema).
- All values as strings — no type inference, no formatting changes.
- UTF-8 encoding with BOM for Excel/Tableau compatibility.
- File naming: consolidated_{date}.csv (e.g., consolidated_2026-03-18.csv).
- Placement: uploaded to a separate SharePoint folder (CSVConsolidated/) or saved locally.

3e. Error Handling and Logging

Scenario Handling
SharePoint API auth failure Retry 3× with exponential backoff, then alert and abort
Empty dated subfolder (no CSVs) Log warning, mark folder as "processed" (nothing to do)
CSV parse error (malformed file) Log error with filename, skip file, continue with remaining files, include skipped files in summary
Network timeout during download Retry individual file download 3×, then skip and log
Upload failure Retry 3×, then save locally and alert
Duplicate folder processing State file prevents re-processing; force-reprocess via CLI flag
Agent crash mid-processing On restart, state file shows "pending" status, agent retries from scratch

Logging:
- Structured logging (JSON format) to both file and stdout.
- Log levels: DEBUG (API calls), INFO (folder detected, consolidation started/completed), WARNING (skipped files), ERROR (failures).
- Log rotation: 7-day retention, 10 MB max per file.

4. Technology Options

Option A: Microsoft Power Automate + Python Script

How it works:
- Power Automate flow triggers on "When a file is created in a folder" in SharePoint.
- Flow calls an Azure Function or webhook that runs the Python consolidation script.
- The 30-minute delay is handled via Power Automate's "Delay" action.

Pros:
- Native SharePoint integration (no auth code to manage).
- Built-in monitoring in the Power Automate portal.
- Low-code trigger setup.

Cons:
- Power Automate has per-action pricing at scale.
- Limited control over retry logic and error handling.
- "When a file is created" trigger can fire per-file, requiring debounce logic.
- Python execution still needs a host (Azure Function, VM, etc.).
- Harder to version-control and test the flow logic.

Option B: Azure Functions + SharePoint API (Microsoft Graph)

How it works:
- A timer-triggered Azure Function runs every 5 minutes.
- Uses Microsoft Graph API with app-only (client credentials) authentication.
- The function contains the full monitor → wait → consolidate → upload pipeline.
- State is stored in Azure Blob Storage or Table Storage.

Pros:
- Serverless — no infrastructure to manage, auto-scales.
- Native Azure AD integration for SharePoint auth.
- Built-in monitoring (Application Insights).
- Cost-effective for periodic polling (consumption plan).

Cons:
- Requires Azure subscription and Azure AD app registration.
- Cold starts can add latency (mitigated by timer frequency).
- More setup overhead for small teams not already on Azure.

Option C: Python Standalone Script with SharePoint SDK

How it works:
- A Python script uses the Office365-REST-Python-Client library or msal + requests for Graph API.
- Runs on any machine (VM, on-prem server, container, developer laptop).
- Scheduled via cron (Linux), Task Scheduler (Windows), or systemd timer.
- State stored in a local JSON file.

Pros:
- Simplest to develop, test, and debug.
- Runs anywhere — no cloud dependency beyond SharePoint itself.
- Full control over every aspect of the pipeline.
- Easy to version-control (single Python file + config).
- No additional Azure services or licensing required.

Cons:
- Requires a host machine that is always on (or scheduled to be on).
- Auth token management is manual (app registration still needed).
- Monitoring/alerting must be set up separately (e.g., email on failure).

5. Recommended Approach

Option C — Python Standalone Script (with Microsoft Graph API)

Justification:

  1. Simplicity: The requirements are straightforward — poll, wait, concat, upload. This does not need serverless infrastructure or a low-code orchestrator. A single Python script is the easiest to write, test, debug, and maintain.

  2. Portability: The script can run on any machine — a VM, a container, a developer's laptop during testing, or an on-prem server. If the team later wants to move to Azure Functions, the core logic (monitor + consolidate) transfers directly.

  3. Full control: Error handling, retry logic, logging, and state management are all explicit in the code. There are no black-box platform behaviors to debug.

  4. Cost: No additional Azure services beyond the existing SharePoint tenant. The compute cost is minimal (a few API calls every 5 minutes).

  5. Tableau compatibility: The script can output the CSV in exactly the format Tableau expects, with no intermediary transformations.

Auth approach: Register an Azure AD app with Sites.Read.All and Files.ReadWrite.All permissions (application-level). Use MSAL for token acquisition. This avoids per-user auth and works unattended.

6. Step-by-Step Implementation Guide

Step 1: Azure AD App Registration

  1. Go to Azure Portal → Azure Active Directory → App registrations → New registration.
  2. Name: SharePoint CSV Agent.
  3. Supported account types: Single tenant.
  4. No redirect URI needed (daemon app).
  5. After creation, note the Application (client) ID and Directory (tenant) ID.
  6. Go to Certificates & secrets → New client secret → copy the secret value.
  7. Go to API permissions → Add permission → Microsoft Graph:
  8. Sites.Read.All (Application) — to list folders and files.
  9. Files.ReadWrite.All (Application) — to download CSVs and upload consolidated output.
  10. Click Grant admin consent.

Step 2: Identify SharePoint Site and Folder Structure

Determine the following values:
- Site URL: e.g., https://contoso.sharepoint.com/sites/DataTeam
- Drop folder path: e.g., Shared Documents/CSVDrops
- Output folder path: e.g., Shared Documents/CSVConsolidated
- Date folder format: e.g., YYYY-MM-DD or MMDDYYYY (configurable)

Step 3: Set Up the Python Environment

python3 -m venv venv
source venv/bin/activate
pip install msal requests pandas

Step 4: Configure the Agent

Create a config.json (or use environment variables):

{
  "tenant_id": "your-tenant-id",
  "client_id": "your-client-id",
  "client_secret": "your-client-secret",
  "site_url": "https://contoso.sharepoint.com/sites/DataTeam",
  "drop_folder": "Shared Documents/CSVDrops",
  "output_folder": "Shared Documents/CSVConsolidated",
  "date_format": "%Y-%m-%d",
  "wait_minutes": 30,
  "state_file": "agent_state.json",
  "log_file": "agent.log",
  "add_source_metadata": false
}

Step 5: Deploy the Script

Copy sharepoint_csv_agent.py and config.json to the host machine.

Step 6: Schedule Execution

Linux (cron):

# Run every 5 minutes
*/5 * * * * cd /opt/csv-agent && /opt/csv-agent/venv/bin/python sharepoint_csv_agent.py >> /var/log/csv-agent-cron.log 2>&1

Linux (systemd timer):

# /etc/systemd/system/csv-agent.timer
[Unit]
Description=SharePoint CSV Agent Timer

[Timer]
OnBootSec=2min
OnUnitActiveSec=5min

[Install]
WantedBy=timers.target

# /etc/systemd/system/csv-agent.service
[Unit]
Description=SharePoint CSV Agent

[Service]
Type=oneshot
WorkingDirectory=/opt/csv-agent
ExecStart=/opt/csv-agent/venv/bin/python sharepoint_csv_agent.py
User=csv-agent

Windows (Task Scheduler):
- Trigger: On a schedule, repeat every 5 minutes.
- Action: Start a program → python.exe with argument sharepoint_csv_agent.py.

Step 7: Verify

  1. Manually create a dated subfolder in the SharePoint drop location.
  2. Upload a few test CSV files.
  3. Run the agent script manually and observe logs.
  4. After 30 minutes (or reduce wait_minutes to 1 for testing), confirm the consolidated CSV appears in the output folder.
  5. Open the consolidated CSV in Tableau and verify all data is present.

7. Deployment and Scheduling Considerations

Idempotency

The agent is idempotent — running it multiple times for the same dated folder produces the same result. The state file prevents duplicate processing. To force reprocessing, use the --reprocess 2026-03-18 CLI flag.

Concurrent Execution

The agent uses a file-based lock (agent.lock) to prevent overlapping runs when using cron. If a previous run is still in progress, the new invocation exits immediately.

Scaling

For most use cases, a single-threaded agent polling every 5 minutes is sufficient. If the volume of files is very large (thousands of CSVs per day), consider:
- Parallel file downloads using concurrent.futures.ThreadPoolExecutor.
- Chunked pandas reads for very large files (chunksize parameter).
- Increasing the VM's memory if consolidated output is multi-GB.

Security

  • Store client_secret in environment variables or a secrets manager (Azure Key Vault, AWS Secrets Manager) — not in config.json in production.
  • The agent host should have network access to login.microsoftonline.com and graph.microsoft.com.
  • Use a dedicated Azure AD app with minimum required permissions.

Backup and Recovery

  • The state file is the only persistent artifact. Back it up or store it in a shared location.
  • If the state file is lost, the agent will re-detect all unprocessed folders on the next run. Already-consolidated output files in SharePoint serve as a secondary indicator of processed folders.

8. Monitoring and Alerting

Log-Based Monitoring

The agent writes structured logs to agent.log. Key log events to monitor:

Event Log Level Meaning
new_folder_detected INFO A new dated subfolder was found
wait_started INFO 30-minute timer started for a folder
consolidation_started INFO Processing began for a folder
consolidation_completed INFO Output CSV uploaded successfully
file_skipped WARNING A CSV could not be parsed
auth_failure ERROR Could not authenticate to SharePoint
upload_failure ERROR Consolidated CSV could not be uploaded

Alerting Options

  1. Email on failure: The script can send an email via SMTP or Microsoft Graph on ERROR-level events.
  2. Health check: Expose a simple HTTP endpoint (or write a heartbeat file) that an external monitor (UptimeRobot, Datadog, etc.) can check.
  3. Cron job monitoring: Use a dead-man's switch service (e.g., Cronitor, Healthchecks.io) — the agent pings the service on each successful run; if no ping arrives within the expected window, an alert fires.
  4. Log aggregation: Ship logs to a centralized system (ELK, Splunk, Azure Monitor) for dashboards and alerts.

Recommended Minimum

  • Set up email alerts for auth_failure and upload_failure.
  • Use a dead-man's switch to detect if the agent stops running entirely.
  • Review logs weekly for file_skipped warnings to catch recurring data quality issues.
