01 System Overview & Architecture

What is VM2?

VM2 ("Varun Malhotra 2") is an automated Chief-of-Staff AI system built on Perplexity Computer for Varun Malhotra, President of Changeis Inc. It runs 12 scheduled tasks that automate:

  • Daily Executive Agenda — Morning briefing with calendar, email triage, and task priorities
  • Federal Opportunity Intelligence — Monitoring DHS APFS and SAM.gov for contract opportunities, RAG-scoring them against Changeis past performance, and generating deep-dive analyses
  • Task Automation — Executing tasks from Todoist (VM2 label) and Gmail (vm2.viru@gmail.com)
  • Compliance & Governance — Nightly audits, weekly digests, monthly integrity checks
  • Business Development — YPO introduction tracking, Todoist auto-helper suggestions

Data Flow Summary

  1. Cron fires → loads required user skills → collects data from external APIs/connectors
  2. Processes data (filtering, RAG scoring, triage) → generates HTML deliverable

  4. Tracks state in JSON files under /home/user/workspace/cron_tracking/

Full task specification 
Run the morning-routine skill: Load skill 'morning-routine' first, then follow its full procedure to generate the Daily Executive Agenda.

DATA COLLECTION (run in parallel):
1. OUTLOOK CALENDAR: Use search_calendar (source: outlook) for today's events

   NOTE: The Todoist connector does NOT support due-date filters. You must fetch broadly and filter in code: parse each task's due.date field, keep only tasks where due.date <= today OR due.date is null (no due date but not completed). Also check for @VEB label.

- Generate the Obsidian Markdown per the skill spec
- Save all three to /home/user/workspace/outputs/
- ALSO save the HTML to /home/user/workspace/VM2-P-Taskers/daily-executive-agenda-YYYY-MM-DD.html

EMAIL DELIVERY:
- Outlook send_email only supports plain-text body (not HTML). Therefore:
  1. Save the HTML brief as an attachment file

09 Global Workflow Rules

These rules apply to EVERY deliverable and EVERY cron

  1. portal.html is the index — NEVER touch index.html. It is an immutable redirect stub.
  2. EVERY deliverable is logged in portal.html — the Deliverables Index is where Varun picks up work.
  3. Context slug required — Every deliverable includes: Session Context (collapsible), Hidden JSON block (with cost_estimate), Continue This Work panel (copy prompt, email VM2, create Todoist), Footer.
  4. Plain HTML only — NO StaticCrypt encryption. Railway nginx basic auth handles password protection.
  5. Triple publish — Push to GitHub, update portal.html, save to Dropbox.
  6. Git config — user.email="varun@changeis.com", user.name="VM2-Perplexity"
  7. Git pull --rebase before every push — prevents conflicts from concurrent cron pushes.
  8. Each task = separate session — Use run_subagent per task for follow-up continuity.
  9. Full memory — Pick up on prior analysis and reasoning, never start from scratch.
  10. Outlook plain-text body — Outlook connector only supports plain text. Always use attachment for HTML.
  11. Todoist connector limit — Max 50 tasks. Always use direct REST API with pagination as primary method.
  12. Delivery log for recurring tasks — Append deliverable link to task description after each execution.

Key Lessons Learned

  • Auto-dive was removed (v3.1) because it created tasks Varun didn't request. The daily brief is INFORMATIONAL ONLY.
  • The vm2-recurring label system (v3.2) prevents catch-up loops for recurring tasks with both VM2 and vm2-done labels.
  • Delivery logs (appended to Todoist task descriptions) provide a running history of recurring task outputs.
  • The Todoist proxy on Railway (/api/todoist/) keeps the API token out of page source.
  • portal.html must insert rows after <tbody id="tableBody"> with proper data attributes for filtering to work.
