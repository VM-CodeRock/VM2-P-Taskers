1. System Overview

VM2 is a fully automated AI task pipeline built on Perplexity Computer. It receives work instructions via email, direct chat, or Todoist, executes them through AI agents, publishes HTML deliverables to a password-protected Railway web portal, and backs up everything to Dropbox and GitHub automatically. As of May 16, 2026, the repo contains 816 deliverables across 14 active scheduled crons.

Core principle: Every task produces a deliverable HTML page. Every deliverable is indexed, hosted on Railway, pushed to GitHub, and synced to Dropbox — automatically, with no manual steps required. Related deliverables are linked via thread tags. Publicly shareable deliverables use a dedicated GitHub Pages repo (Tier 2).

2. Component Architecture

2.1 Portal & Index

Rule: portal.html is the sole deliverables index. index.html is an immutable ~229-byte redirect stub pointing to portal.html. No cron or session may modify index.html content. The nightly compliance audit enforces this.

Portal columns (v4.1+): portal.html has 5 columns plus optional thread tag attribute:

Column / AttributeClass / AttrSourceDescription
Createdcol-datedata-dateDate the deliverable was first published
Titlecol-title—Linked title opening the deliverable; may include secondary public share link in notes
Typecol-typedata-typeBadge-styled category (Deep Dive, Monitoring, Brief, etc.)
Updatedcol-updateddata-updatedDate of last revision (same as Created if v1)
Linkcol-link—Arrow icon linking to the deliverable
Thread Tag v4.3data-threadThread slugOptional attribute on <tr> — links deliverable to a cross-deliverable thread. Empty string if no thread.

Default sort: descending by data-updated so the most recently revised deliverables appear first.

2.4 Gmail Task Pipeline

  • Inbox: vm2.viru@gmail.com
  • Monitored hourly 7 AM – 10 PM EDT daily (including weekends) by the vm2-gmail-tasks skill cron (0 0-2,11-23 * * *)
  • State is dual-written to two paths kept in sync:
    • PRIMARY: /home/user/workspace/cron_tracking/gmail_intake/state.json
    • FALLBACK: /home/user/workspace/cron_tracking/6a1f1c3b/state.json
  • Emails with "VM2" in subject/body are parsed as task instructions
  • Each task spawns a dedicated Perplexity Computer session
  • Completed emails are labeled vm2-done and a reply is sent with the deliverable link
  • If Varun replies to an email with further instructions, those instructions are acted upon at the next hourly intake
  • Thread tag extraction (v4.3): Subject lines containing [thread:slug] are parsed; the slug is injected into the deliverable's data-thread portal attribute and "thread" context slug field automatically
  • send_email retry pattern (v4.3): When a subagent's send_email call fails, the parent session retries via a direct call_external_tool call

ID verification rule: When a cron escalation or subagent reports a task ID or email ID, always verify the reported ID/content against a REST fetch before processing. 5. GitHub & Railway Hosting

5.1 Push Workflow

# Standard deliverable push
git add <filename>.html portal.html
git commit -m "Deliverable: <slug> · VM2 · YYYY-MM-DD"
git pull --rebase origin main   # resolve any concurrent pushes
git push origin HEAD
# Railway auto-deploys within ~60 seconds

5.2 Non-Fast-Forward Recovery (v4.5)

Resolution for non-fast-forward push failures: git pull --rebase origin main then git push origin HEAD. Never use force-push — risks overwriting concurrent deliverables.

5.3 HTML Requirements

  • All HTML must be plain HTML — no StaticCrypt encryption
  • Railway nginx basic auth protects all pages server-side
  • Deliverables should include a "Continue This Work" footer panel with context for follow-up sessions
  • Context slug at write-time (v4.5): Slug must be added at deliverable creation time, not via post-hoc retrofit.
  • Daily Brief date fix (v4.6): Combined Daily Brief builder script must use runtime date, not hardcoded 2026-05-13 date.

9.3 SAM Stuck-Run Cleanup Procedure (v4.6 — NEW 5/15)

Stuck-run cleanup: If the SAM SQLite database shows a run with status='in_progress' but no process owns it (lockfile absent or PID dead), run: UPDATE runs SET status='failed' WHERE status='in_progress'; before the next scheduled run. Failure to clean up will cause the monitor to skip execution, believing a run is already active.
