1. System Overview

VM2 is a fully automated AI task pipeline built on Perplexity Computer. It receives work instructions via email, direct chat, or Todoist, executes them through AI agents, publishes HTML deliverables to a password-protected Railway web portal, and backs up everything to Dropbox and GitHub automatically. As of April 22, 2026, the repo contains 586+ deliverables across 14 active scheduled crons.

v4.2 headline changes (Apr 22, 2026): (1) Opportunity Deep Dive Monitor cron 6f5371ab deleted — all deep dives are now email-driven only. (2) Daily Brief "Deep Dive" buttons are now mailto: links to vm2.viru@gmail.com — no more automatic Todoist task creation. (3) SAM.gov Monitor rebuilt as SQLite-backed pipeline using authenticated api.sam.gov (previous unauthenticated endpoint was returning 0 results for 18 days). (4) New vm2-done Todoist label (id 2183604438) created to satisfy compliance Rule 5. (5) 30-day backfill seeded 1,452 opportunities into the new SAM SQLite DB.

2.5.1 Recurring Task Model (v3.2)

Recurring Todoist tasks (e.g., daily SAM checks, weekly reviews) use a three-label system:

Label ComboBehavior
VM2 onlyOne-shot task. Execute when due/overdue. Add vm2-done on completion.
VM2 + vm2-recurringExecute ONLY if due.date == today (exact match). Skip overdue — prevents catch-up loops. vm2-done label is ignored for eligibility.
VM2 + vm2-done (no recurring)Completed one-shot. Skipped permanently.

Delivery Log: After each recurring execution, the cron appends a timestamped link to the task's description in Todoist:

── VM2 Delivery Log ──
✓ 2026-04-06 — https://vm2-p-taskers-production.up.railway.app/faa-daily-sam-2026-04-06.html
✓ 2026-04-07 — https://vm2-p-taskers-production.up.railway.app/faa-daily-sam-2026-04-07.html

This builds a running history of all deliverables directly on the task — no portal searching needed.

State tracking: /home/user/workspace/cron_tracking/todoist_vm2/recurring_state.json records {task_id: last_execution_date} to prevent double-runs within the same day (hourly cron). The new cron (64b6a5cc) reads from both the legacy path (cron_tracking/59996e17/recurring_state.json) and the current cron_tracking/todoist_vm2/ path.

2.8 Operational Policy — No-Confirmation

All actions are pre-approved. No confirm_action prompts are needed for email replies, deep dives, task execution, or remediation. VM2 proceeds automatically.

User instruction (verbatim):

"Everything is approved for next steps in the future. I do not need these pending approval requirements. If I have responded to an email with further instructions, it should be acted upon at the next intake."

This policy covers:

  • Gmail reply sending — replies go out without confirmation prompts
  • Deep dive deliverable creation and publishing
  • Todoist task execution and vm2-done labeling
  • Deliverable Integrity Audit auto-remediation
  • Any follow-on actions triggered by email reply instructions

4.2 Sync Mechanism

The nightly vm2-wrapup skill (v2.1+) performs a full diff sync:

  1. Lists all .html files in /VM2-main-folder/VM2-P/Taskers/ recursively
  2. Lists all deliverable HTML files in the GitHub repo
  3. Computes the diff (GitHub − Dropbox)
  4. Uploads all missing files to the correct categorized subfolder
  5. Reports N synced, N already present in the compliance report

5. GitHub & Railway Hosting

5.1 Push Workflow

# Standard deliverable push
git add <filename>.html portal.html
git commit -m "Deliverable: <slug> · VM2 · YYYY-MM-DD"
git pull --rebase origin main   # resolve any concurrent pushes
git push origin main
# Railway auto-deploys within ~60 seconds
Conflict resolution: Multiple crons may push concurrently. Always use git pull --rebase before pushing. If rebase fails, the cron escalates to the foreground session for manual resolution.

5.2 HTML Requirements

  • All HTML must be plain HTML — no StaticCrypt encryption
  • Railway nginx basic auth protects all pages server-side
  • Deliverables should include a "Continue This Work" footer panel with context for follow-up sessions

7.6 Skip Rules (NO-GO Categories)

File: /home/user/workspace/cron_tracking/6f5371ab/skip_rules.json (retained even though the cron was deleted — still used by Gmail TYPE A handler).

When a deep dive is requested via email, the Gmail intake applies these 12 NO-GO rules to decide whether to generate the deliverable. If any rule matches, the intake replies with a one-paragraph reason and takes no further action; Varun can reply "Override skip rule" to force generation.

Example categories: set-aside small-business-only procurements, sole-source to a named incumbent, NAICS outside Changeis's 52-code priority list, geographic mismatch, dollar-threshold too low relative to Changeis minimums, closed/archived opportunities, duplicative with a deep dive already delivered within 30 days.
