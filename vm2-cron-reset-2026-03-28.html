Full architecture overhaul — 12 new crons, compliant from day one
March 28, 2026
Executive Summary

A full audit of all VM2 crons revealed 0 out of 16 deliverables were compliant with the post-overhaul architecture rules. The root cause: every active cron was created in a prior session before the skill overhaul, and those sessions had baked-in pre-overhaul behavior that no patch could fix.

  • Why the reset was needed: Old crons referenced pre-overhaul skill versions — no context slug, no cost estimate, no "Continue This Work" panel, no architecture rule embedding. The compliance audit detected 0/16 passing deliverables.
  • What changed: All 12 old crons were stopped. 12 new crons were created in a single session with updated skill versions. Every cron's task description now contains the 7 immutable architecture rules.
  • New system design: Four-tier architecture (Morning Sequence / Hourly Pipelines / Periodic Maintenance / Temporary). Staggered minutes (:15/:29/:45) prevent hourly collisions. Todoist Task Execution is now hourly — same cadence as Gmail.
  • One cron preserved: CCC Cardio Tennis (20aefc72) was left in its original session — it is a simple browser task unaffected by deliverable rules and does not produce HTML deliverables.

Hourly Pipeline Timeline — :15 / :29 / :45 Stagger

Three hourly crons run sequentially within each hour window (8 AM–7 PM EDT, M–F), staggered to prevent session collisions and allow each cron's spawned sessions to complete before the next scan.

Why staggered? Each cron spawns sub-sessions that take 5–15 minutes. Staggering by 14–16 minutes ensures no two parent crons are fighting for resources simultaneously, and task deduplication files are written before the next scan.
Key Changes vs. Old System
  • Todoist Task Execution is now HOURLY — previously ran on a different cadence, now aligned with Gmail at :15 past each hour for symmetry and predictability.
  • Auto-Helper redesigned — old version only scanned VM2-labeled tasks. New version scans all Todoist tasks and suggests which non-VM2 tasks AI could help complete.
  • Architecture rules baked into every cron — each cron's task description contains all 7 immutable rules, ensuring any session that runs the cron is immediately aware of them without relying on skill version state.
  • Staggered minutes (:15/:29/:45) — prevents simultaneous execution collisions in the hourly window.
  • Nightly audit now detects stale crons — enhanced to identify crons still running from old sessions (pre-reset), which appear in memory but whose sessions may be dead or non-compliant.
  • Single-session creation — all 12 crons were created in one session, ensuring consistent skill versions and rule embedding throughout.

Architecture Rules — 7 Immutable Rules

These rules are now embedded in every new cron's task description. Any session running a cron sees them immediately.

  • portal.html = sole deliverables index; index.html = immutable redirect stub — never edit index.html
  • Every deliverable gets a context slug: JSON block + visible Session Context panel + Continue This Work panel + cost estimate
  • No StaticCrypt encryption — plain HTML only; Railway handles authentication
  • Version convention: slug-YYYY-MM-DD-v2.html with prior_versions chain in the JSON
  • Deliverables ARE the memory — sessions are ephemeral, deliverables are permanent
  • Each task spawns a separate Perplexity Computer session for follow-up work
