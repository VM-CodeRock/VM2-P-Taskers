Section 1 · Orientation

Why this guide and where you sit today

You already use Claude Code daily for proposal analysis, RAG over solicitation libraries, opportunity deep-dives, and HTML deliverable generation. The leap from power user to enterprise-grade operator is not about prompts — it is about architecture: persistent memory, reusable skills, governed tools, and autonomous loops that survive a session reset.

Maturity model

Level 1
Prompter
  • Ad-hoc prompts
  • One CLAUDE.md, if any
  • Manual copy/paste
  • Re-explains context each session
Level 2
Power user
  • Project CLAUDE.md
  • A few slash commands
  • Plan mode for big changes
  • ← you are here
Level 3
Architect
  • Skills library + path-scoped rules
  • Specialized subagents
  • SessionStart hooks
  • MCP servers vetted & checked in
Level 4
Autonomous operator
  • Headless cron / GitHub Actions
  • Self-updating MEMORY.md
  • Eval & regression suite for skills
  • Audited, reproducible deliverables
Goal of this guide
Move you from Level 2 → Level 4 in ~30 days, using your existing VM2-P-Taskers / Changeis daily-brief workflows as the live laboratory.
Section 2 · Framework

The six pillars of an enterprise-grade Claude Code setup

Pillar 1

Memory

Durable instructions (CLAUDE.md) + auto-learned notes (MEMORY.md) + path-scoped rules. Survives every /clear.

Pillar 2

Context

Pillar 3

Skills

Pillar 6

MCP & Governance

Connectors checked into .mcp.json. Least-privilege tools, sandbox, audit logs.

Pillar 1 · Memory

Memory architecture: CLAUDE.md, MEMORY.md, and rules

Two complementary systems load on every session: CLAUDE.md (you write the rules) and auto-memory MEMORY.md (Claude writes its learnings). Master the split and you stop re-teaching every conversation.

The four scopes you should actually use

ScopePathWhat goes here
Managed policy/Library/Application Support/ClaudeCode/CLAUDE.md (mac)Org-wide standards: CMMC reminders, "never paste customer PII into prompts," approved model list.
Project./CLAUDE.md or ./.claude/CLAUDE.mdProject architecture, build commands, deliverable format ("HTML files named slug-YYYY-MM-DD.html, push to VM-CodeRock/VM2-P-Taskers").
User~/.claude/CLAUDE.mdYour personal preferences across projects: bullet-point summaries, Changeis style guide pointer, preferred Python/Node versions.
Local./CLAUDE.local.md (gitignored)Sandbox URLs, test creds, "currently debugging Railway deploy."
Hard rules
  • Keep each CLAUDE.md under 200 lines. Specificity beats volume.
  • Move multi-step procedures into a skill; move file-pattern rules into .claude/rules/ with paths: frontmatter.
  • Use @imports (5-hop max) to compose: @~/.claude/changeis-style.md in your project file.

Auto-memory: let Claude take the notes

Lives at ~/.claude/projects/<project>/memory/MEMORY.md. First 200 lines / 25 KB load every session; topic files (debugging.md, api-conventions.md) load on demand. Trigger it by saying things like "remember that the FAA APEX deep-dive uses unlimited RAG matching, top_n: 50."

# /memory                  → list all loaded files, toggle auto-memory
# claude --version          → confirm v2.1.59+ for auto-memory
# claude --add-dir ../shared → load CLAUDE.md from sibling dir

Path-scoped rules — the under-used superpower

Rules in .claude/rules/ with paths: frontmatter only fire when Claude touches matching files. Perfect for "when editing anything under proposals/**, follow Changeis style guide."

---
paths:
  - "proposals/**/*.docx"
  - "deliverables/**/*.html"
---
# Changeis Deliverable Rules
- Cover bar #1B2A4A, H1 #002060, body Georgia 11pt
- Tables: header #1B2A4A bg / white text, alt rows #F2F6FA
- Footer: "Changeis Confidential — [Type] — [Month Year]"
Quick win for you
Symlink your existing changeis-style skill into ~/.claude/rules/changeis-style.md. Now every project automatically inherits brand standards without you remembering to mention them.
Pillar 2 · Context

Context engineering: spend tokens like a CFO

Long sessions degrade. Output quality drops as the context window fills. Treat context as a budget — and offload aggressively to files and subagents.

The four moves

  1. Plan in files, not chat. Maintain a PLAN.md or TODO.md Claude updates as it works. When context resets, Claude re-reads the file. This is the single highest-ROI habit.
  2. Pre-empt auto-compaction. Run /compact preserve the current implementation plan, all file paths, and the deep-dive RAG results before Claude does it for you. You control what survives.
  3. Use /clear at task boundaries. Between unrelated tasks (RFI response → daily brief → Motus dashboard), clear is faster and cleaner than letting noise accumulate.
  4. Delegate exploration to subagents. Have a Researcher subagent read 50 files and return a 200-token summary. The parent never sees the noise.

Anatomy of a token-efficient session

PhaseWhat you doWhy
StartOpen in project root so all CLAUDE.md ancestors loadFree instructions, zero re-typing
ExploreSend subagent to map codebase / read solicitation PDFNoise stays in subagent's window
PlanPlan mode (Shift+Tab twice). Save plan to PLAN.mdForces explicit plan; persists across resets
ExecuteReference plan: "execute step 3 of PLAN.md"Tight, scoped prompts; less drift
Compact/compact preserve plan + file pathsYou decide what context survives
EndTell Claude what to remember; it writes to MEMORY.mdNext session starts smarter
Pillar 3 · Skills

Skills: your reusable, enterprise-grade IP

Skills are the new format for everything reusable — they replaced legacy slash commands. Each skill is a folder with a SKILL.md (YAML frontmatter + markdown body). Claude can invoke them autonomously, or you type /skill-name.

Skill anatomy

~/.claude/skills/changeis-deep-dive/
├── SKILL.md          # entry point (frontmatter + instructions)
├── prompts/
│   ├── incumbent.md  # FPDS / USAspending research prompt
│   └── go-no-go.md   # bid decision template
├── templates/
│   └── deep-dive.html.j2
└── scripts/
    └── fetch_fpds.py
---
name: changeis-deep-dive
description: Run a Changeis opportunity deep-dive — RAG match (top_n=50),
  FPDS incumbent, USAspending, capability matrix, Go/No-Go. Use when
  the user asks for a deep dive, opportunity analysis, or bid assessment.
disable-model-invocation: false   # let Claude auto-pick when relevant
allowed-tools: Read, Write, Bash(python:*), WebFetch
---

# Changeis Deep-Dive Skill v2.1
1. Read solicitation from $ARGUMENTS or ./solicitation.pdf
2. RAG match against ~/changeis/capability-library (top_n=50)
3. Run scripts/fetch_fpds.py for incumbent data
4. Render templates/deep-dive.html.j2 → deliverables/
5. Push to VM-CodeRock/VM2-P-Taskers, update portal.html

The skills you should build first

SkillReplacesInvocation
changeis-deliverableRe-stating brand colors every timeAuto, on any HTML/DOCX/PPTX
changeis-deep-dive22-step RFI workflow in your head/changeis-deep-dive {opp-id}
changeis-daily-briefManual APFS + SAM.gov mergeCron / headless
portal-publishHand-editing portal.html rowsAuto, after any deliverable write
compliance-scrubManual CMMC / NIST 800-171 review/compliance-scrub PROPOSAL.docx

Skill governance
Check skills into VM-CodeRock/VM2-P-Taskers/.claude/skills/. Tag versions (v2.1.0). Maintain an EVAL.md per skill with 3–5 reference inputs and expected outputs — run before merging changes.
Pillar 4 · Subagents

Subagents and plan mode

Subagents are specialists with their own system prompt, tool list, and 200K context. They do work in isolation and return a small summary. Per Anthropic's own guidance: "sub-agents work best when they just look for information and provide a small amount of summary back to the main thread."

Define a subagent

# .claude/agents/incumbent-researcher.md
---
name: incumbent-researcher
description: Pulls FPDS contract history, USAspending obligations,
  and award trends for a given solicitation or contractor.
tools: WebFetch, Read, Bash(python:*)
---
You are an expert federal contracting researcher. Given a solicitation
ID or contractor name, return: incumbent contractor, contract value,
period of performance, key personnel, and 3 recompete risk signals.
Always cite sources with URLs. Output JSON, ≤500 tokens.

The "Explore → Plan → Execute" loop

  1. Explore (subagents only). Researcher reads docs, code, logs. Returns summaries.
  2. Plan (main agent in plan mode, Opus-class model). Reads summaries, writes PLAN.md. Forces approval before any code/file change.
  3. Execute (main agent or executor subagent). Follows plan. Reports back with diffs.
Pattern: the Capture Capsule
For each new opportunity: spawn 3 subagents in parallel — incumbent-researcher, rag-matcher, compliance-scanner. Wait for all three. Parent agent composes the deep-dive HTML from the three summaries. Total token cost ~30% of doing it serially in one window.
Pillar 5 · Hooks & Commands

Hooks, slash commands & output styles

The hooks worth setting up today

EventUse case for youAction
SessionStartInject today's APFS top-5 + active Todoist vm2-opp tasksBash script writes to stdout → becomes context
UserPromptSubmitAuto-append "follow Changeis style" if prompt mentions deliverablePre-process prompt before Claude sees it
PostToolUse (Edit)Lint HTML, run portal.html schema checkBlock on exit code 2 with stderr message
PreToolUse (Bash)Block rm -rf, git push --force, anything outside workspaceExit 2 → Claude sees error, retries differently
StopRun nightly wrap-up audit (mirrors your vm2-wrapup skill)Verify Dropbox + GitHub + portal sync

Slash commands — now invocable by Claude

Since Claude Code v1.0.123 the SDK exposes a SlashCommand tool: Claude itself can decide to run /your-command. This turns slash commands from human shortcuts into callable APIs for the agent. Combined with skills, they form your reusable workflow library.

Output styles for different audiences

  • Default — daily work.
  • Explanatory — when learning a new MCP server or library.
  • Custom "Executive" — bullet-point only, ≤120 words, action-oriented (matches your bullet-point preference). Ship as ~/.claude/output-styles/executive.md.
Pillar 6 · MCP & Governance

MCP servers: power, with guardrails

MCP is how Claude Code reaches your real systems — Todoist, Dropbox, GitHub, Jira, Outlook. Every server you add expands the attack surface. Anthropic's enterprise guidance: centralize MCP config in .mcp.json, check it into the repo, so the whole team gets the same vetted set.

Recommended MCP set for your stack

ServerUseRisk & mitigation
githubPush deliverables to VM2-P-Taskers, open PRsUse a fine-grained PAT, scoped to the repo, expires 90d
filesystem (Dropbox via local mount)Backup HTML to V M2/VM2-main-folder/VM2-P/Restrict to that path; deny via permissions.deny
todoistRead vm2-opp queue, mark vm2-doneRead-write only on those labels
fetch / playwrightSAM.gov, FPDS, USAspending pullsCache aggressively; rate-limit
postgres / sqlite (optional)RAG vector store of solicitation libraryRead-only role for Claude

The .mcp.json + settings split

// .mcp.json (checked into repo)
{
  "mcpServers": {
    "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] },
    "todoist": { "command": "uvx", "args": ["mcp-todoist"] }
  }
}

// .claude/settings.local.json (gitignored)
{
  "permissions": {
    "deny": ["Bash(rm -rf:*)", "Bash(git push --force:*)"],
    "allow": ["Skill(changeis-*)", "Bash(python:*)"]
  },
  "sandbox": { "enabled": true }
}
Section 3 · Autonomy

Autonomous ops: headless mode + GitHub Actions

Once your skills are stable and your CLAUDE.md is tight, Claude Code stops being a chat tool and becomes a CI worker. This is where you mirror — and exceed — what your VM2 cron stack does today.

Three autonomy patterns

Pattern A

Local cron + headless

claude -p "/changeis-daily-brief" --output-format json at 06:00 ET. Pipes JSON to your existing publisher.

Pattern B

GitHub Actions

anthropics/claude-code-action on PR, issue comment @claude, or schedule. Reviews proposals, generates summaries, opens PRs.

Pattern C

SDK pipeline

Pipe data in, JSON out. cat solicitation.txt | claude -p "/deep-dive" --output-format json | jq … — composable in any pipeline.

GitHub Actions skeleton (proposal review)

name: Proposal Review
on:
  pull_request:
    paths: ['proposals/**']
permissions:
  contents: read
  pull-requests: write
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: |
            /compliance-scrub
            /changeis-style-check
            Comment on the PR with red-flag findings only.
          claude_args: --max-turns 8
CI safety checklist
  • Least-privilege tokens; never echo secrets
  • Redact outputs in PR comments
  • Human-in-the-loop on merge for any AI-authored change
  • Pin action versions; review changelogs before bumping
  • Fixed-cost subscription for predictable spend
Section 4 · Apply to Changeis

Patterns mapped to your actual deliverables

Daily brief (APFS + SAM.gov merge)

  1. SessionStart hook pulls top APFS + new SAM.gov hits into context.
  2. Skill changeis-daily-brief renders HTML using path-scoped style rules.
  3. Skill portal-publish appends row to portal.html, pushes to VM-CodeRock/VM2-P-Taskers, mirrors to Dropbox.
  4. Headless cron at 06:00 ET. Output JSON consumed by your existing email automation to varun@changeis.com.

Opportunity deep-dive (your v2 spec)

  1. Plan mode opens. Claude proposes the deep-dive structure.
  2. Three subagents fire in parallel: incumbent-researcher, rag-matcher (top_n=50 against your library), capability-matrix.
  3. Parent composes Go/No-Go using changeis-deep-dive skill template.
  4. Hook on Stop validates schema (data-date, badge classes), pushes to portal.

Proposal / RFI response

  1. Path-scoped rule under proposals/** auto-loads CMMC / NIST 800-171 / FAR clauses checklist.
  2. Skill compliance-scrub on demand: flags missing clauses, undefined acronyms, page-limit overruns.
  3. GitHub Action runs same skill on every PR to proposals/ — comments findings inline.
  4. Final pass: subagent red-team-reviewer with stricter system prompt, returns 5 weakest claims.
Reusability win
Every skill above lives in VM-CodeRock/VM2-P-Taskers/.claude/skills/. When you start V2M2 (OpenClaw) or any new repo, you symlink the skills directory and the entire workflow comes with you — no reconfiguration.

Section 6 · Enterprise-Grade Checklist

Are you actually Level 4?

  • Project + user CLAUDE.md exist, both under 200 lines, no contradictions
  • Auto-memory enabled and Claude has written ≥3 entries to MEMORY.md you didn't ask for
  • ≥1 path-scoped rule firing only on relevant files (verify with /memory)
  • ≥3 skills in .claude/skills/, each with frontmatter description Claude can match
  • ≥1 skill has an EVAL.md and a version tag
  • .mcp.json committed; settings.local.json gitignored with permissions.deny entries
  • ≥1 specialized subagent in .claude/agents/ (researcher / reviewer / scanner)
  • SessionStart hook injects live state (today's tasks, open PRs, active opps)
  • PreToolUse hook blocks destructive commands
  • One headless workflow runs unattended (cron or GitHub Actions)
  • Sandbox enabled in managed settings; secrets never in CLAUDE.md
  • Plan mode is your default for anything multi-file or risky
  • You proactively /compact with directives instead of letting auto-compact run
  • Skills + rules + agents are checked into the same repo as the work — reproducible by any teammate
Score yourself
13–14 → Level 4 autonomous operator. 9–12 → Level 3 architect. 5–8 → Level 2 power user (today). <5 → Level 1.
