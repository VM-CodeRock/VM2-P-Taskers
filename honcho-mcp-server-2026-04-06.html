1. Architecture Overview

The Honcho MCP Server acts as a thin translation layer between Perplexity Computer and the Honcho conversational memory platform. It exposes 6 MCP tools that give every Perplexity session access to persistent, cross-session memory — eliminating context loss from compaction.

ComponentRoleTechnology
Perplexity ComputerAI assistant (any session)MCP Client (Streamable HTTP)
Honcho MCP ServerProtocol bridge + tool hostPython FastMCP on Railway
Honcho API (v3)Memory storage + reasoningREST API + Dialectic Engine
Design Principle: The server is stateless — all memory lives in Honcho. If Railway restarts or the server redeploys, nothing is lost. Honcho's Dialectic engine continuously reasons over stored messages to build a rich, evolving user model.
2. Session Lifecycle

Every Perplexity Computer session follows a three-phase pattern. This is automated via a global memory instruction — no manual intervention required.

Phase 1

Start

Tool: bootstrap_context

Pulls curated memory brief: user profile, representation, peer card, and recent conclusions.

Phase 2

Work

Tool: store_memory

Stores new facts, preferences, decisions, and project state as they emerge during the session.

Phase 3

End

Tool: closeout_session

Writes session summary, key decisions, new preferences, and open items back to Honcho.

Result: The next session's bootstrap picks up exactly where the previous session left off. Compaction no longer causes rework or token waste.
3. MCP Tool Reference

bootstrap_context

When: First call every session

Retrieves curated user profile, working representation, peer card, and recent conclusions from Honcho. Accepts optional search_query to focus on a specific topic and max_conclusions to control volume.

Parameters: peer_id, max_conclusions, search_query

store_memory

When: When a durable fact is learned

Stores facts, preferences, project state, corrections, or decisions. Creates a dedicated Honcho session, adds the peer, and writes the tagged message. Honcho's engine auto-reasons over it.

Parameters: content (required), memory_type, peer_id

lookup_memory

When: Need a specific past fact

Semantic search over Honcho's derived conclusions. Returns content, observer/observed IDs, session, and timestamp.

Parameters: query (required), top_k

search_messages

When: Need raw past interactions

Hybrid text + vector search over all stored messages. Falls back to workspace-level search if peer-level fails.

Parameters: query (required), peer_id, limit

ask_honcho

When: Need synthesized insight

Natural-language Q&A via Honcho's Dialectic API. Reasons over all stored knowledge to answer questions about the user that no single lookup could provide. Supports reasoning levels: minimal through max.

Parameters: question (required), peer_id, reasoning_level

closeout_session

When: Last call every session

Writes session summary, decisions, preferences, and open items. Also stores key items as direct conclusions for fast retrieval in future bootstraps.

Parameters: summary (required), key_decisions, new_preferences, open_items, peer_id

7. Automation Setup

The bootstrap/closeout lifecycle is automated via two mechanisms that reinforce each other:

Perplexity Global Memory

A persistent instruction stored in Perplexity's built-in memory tells every new session to:

1
Call bootstrap_context at the start of every session
2
Call store_memory proactively during the session for durable facts
3
Call closeout_session at the end (or on "wrap up", "done", "close out")

Honcho Self-Reinforcing Protocol

The same workflow protocol is stored in Honcho itself. When bootstrap_context runs, it returns this instruction as part of the user's preferences — ensuring the protocol persists even if Perplexity's memory is cleared.

No manual action needed. New sessions automatically bootstrap and close out. The dual-storage approach (Perplexity memory + Honcho memory) provides redundancy — the protocol survives even if one system is reset.

Key Implementation Details

  • Async pattern: All Honcho calls use .aio accessors (peer.aio.context(), session.aio.add_messages()) to avoid blocking the event loop. Peer and session objects are obtained via await honcho.aio.peer() and await honcho.aio.session().
  • Lazy initialization: The Honcho client is created at server startup, but the first API call (workspace creation, peer get-or-create) is deferred to the first tool invocation. This prevents the MCP handshake from hanging on Honcho API latency.
  • Error handling: Every tool catches HonchoAuthError explicitly and returns a clear message pointing to the HONCHO_API_KEY environment variable — no opaque "Unauthorized" errors.
10. Troubleshooting
IssueResolution
Honcho authentication failedVerify HONCHO_API_KEY in Railway. Get a fresh key at app.honcho.dev/api-keys.
Connector validation fails in PerplexityURL must be https://your-url/mcp. Transport: Streamable HTTP. Auth: None.
404 errors from Honcho APIEnsure honcho-ai≥2.1.0 in requirements.txt (not honcho-core, which uses deprecated v2 paths).
Bootstrap returns empty profileSend store_memory calls first to seed Honcho with data. The reasoning engine needs input to work with.
Railway deploy failsCheck Railway logs. Usually a missing env var or Docker build timeout.
MCP handshake not completingTest with: curl -X POST -H 'Content-Type: application/json' -H 'Accept: application/json, text/event-stream' -d '{...initialize...}' https://your-url/mcp
Tools not auto-triggering in new sessionsRe-store the global memory instruction. Say: "Remember that I use Honcho Memory for persistent cross-session context..." (see Section 7).
