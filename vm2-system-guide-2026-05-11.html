What VM2 Is
  • Your personal AI operating system built on Perplexity Computer
  • Handles: research, competitive intel, opportunity analysis, daily briefs, writing, scheduling, document creation
  • Everything produces a deliverable HTML page hosted on Railway, indexed in your Deliverables Portal

Three Ways to Task VM2
Option 1

Direct Chat

  • Open Perplexity Computer and type your request
  • Deliverable gets pushed to GitHub + Railway automatically
Best for: immediate, interactive work where you want to guide the analysis
Option 2

Email Pipeline

  • Send email to vm2.viru@gmail.com with "VM2" in subject or top of body
  • Add [thread:slug] in the subject to tag it to a thread
  • Hourly cron picks it up (7 AM – 10 PM, every day including weekends)
  • You get a reply with the deliverable link when done
Best for: fire-and-forget tasks, delegating while you're in meetings
Option 3

Auto-Helper (Todoist)

  • Daily at 7 AM, VM2 scans your full Todoist backlog (~700 tasks via REST API) and shows tasks it can handle
  • Open the Auto-Helper page on Railway and click "Task VM2" on any task
  • This sends an email that triggers the pipeline above
Best for: working through your backlog systematically
Public Sharing Pattern (new v2.4)

When a deliverable needs to be shared publicly (family, external collaborators, public demos), VM2 creates a dedicated Tier 2 public GitHub Pages repo — separate from the auth-protected Railway portal.

1
Trigger: You request a publicly shareable URL, OR the deliverable is for a non-Changeis audience (family, external collaborators, public-facing tools). Examples: apps, games, educational tools, shared research.
2
VM2 creates the public repo: gh repo create vm-coderock/{slug}-share --public, pushes the HTML as index.html, and enables GitHub Pages. The public URL is https://vm-coderock.github.io/{slug}-share/.
3
The original Tier 1 row in portal.html stays unchanged. The public share URL is added as a secondary link in the title or notes column. Both versions coexist.
First instance (5/11): Mahjong Tutor — vm-coderock.github.io/mahjong-tutor-share/. All BD/proprietary deliverables remain Tier 1 only.
Thread Tag System (v2.3)

Threads group related deliverables into named topic clusters — without changing how existing deliverables or workflows operate. A thread tag is a lightweight slug (e.g., faa-apex) that binds deliverables together across sessions.

1
To tag a new email task to a thread: include [thread:slug] anywhere in the Gmail subject line. Example: VM2 - [thread:faa-apex] Post-event debrief APEX Industry Day. The intake cron automatically extracts the tag and injects it into the deliverable and portal row.
2
To tag a Todoist task to a thread: apply the vm2-thread label and put thread:slug as the first line of the task description.
3
To view all deliverables in a thread: open the thread's aggregator page, e.g., faa-apex-thread.html. These pages load portal.html client-side and filter by data-thread — always up to date.
Active threads (5 with aggregator pages): faa-apex, faa-atlas, faa-oig-47238, mkt-aic, oasis-plus. Active thread tags (9 total): faa-apex, ydc-golf, exec-comp, mkt-aic, faa-oig-47238, faa-atlas, oasis-plus, apex-framework, wsr-madison.
How to Order a Deep Dive (v2.2+ — Email-Only)
1
Open the Combined Daily Brief (arrives ~10 AM weekdays) — you'll see STRONG and MODERATE opportunities as cards.
2
On any card you want analyzed, click Request Deep Dive. This opens a pre-filled email to vm2.viru@gmail.com with the opportunity metadata already filled in.
3
Add anything in the Notes field if you want to scope the analysis, then send.
4
At the next hourly Gmail intake, VM2 classifies it as TYPE A, applies skip rules, and either (a) generates the full deep-dive deliverable and replies with the Railway link, or (b) replies explaining why it skipped (reply "Override skip rule" to force it).
Why email? Because the Gmail intake is the one pipeline we trust for deterministic processing. This stops the old problem of automated deep dives getting run on opportunities you never asked for.
How to Follow Up on Any Deliverable
1
Open any deliverable on Railway
2
Scroll to the bottom — every page has a "Continue This Work" panel
3
Choose one of three options:
  • Copy Follow-Up Prompt → paste into a new Perplexity Computer session. The new session reads the full context from the deliverable URL — all prior reasoning, findings, dead ends, and open questions.
  • Email to VM2 → sends a follow-up email that the pipeline picks up with full context
  • Create Todoist Follow-Up → adds a follow-up task to your Todoist with vm2 label
4
The new session reads the original deliverable and picks up exactly where the last session left off — no starting from scratch, no lost reasoning

Compliance Slug Regression — Ops Procedure (new v2.4)

When the Nightly Compliance Audit (cron 6494d39d) flags deliverables with missing id="vm2-context-data" slug blocks, use the inline retrofit procedure:

1
Root cause check: Cron subagents couldn't load changeis-style / vm2-wrapup skills (load_skill returned "No space skill found"). The parent agent must handle the retrofit inline.
2
Use the inject_slug.py template: Read each affected deliverable's filename and date, derive the slug from deliverable-specs.md per-file metadata, generate the canonical JSON block, and inject it before </body> tag.
3
Commit and push: git add -A && git commit -m "Compliance: inject missing vm2-context-data slugs (YYYY-MM-DD)". Update Dropbox for retrofitted files.
5/11 instance: 7 files retrofitted — ai-daily-newsletter, browser-use-research-sam-gov, faa-daily-on-sam, faa-odra-recent-decisions, g2x-fedciv-review, gsa-interact-weekly, hermes-agent-use-cases-analysis (all 2026-05-11). Commit c93d819.
Known Issues (v2.4)
These are material pending bugs preserved across refresh cycles. They do not block operation but should be resolved.
  • UTC Rollover Stand-Downs (cron 64b6a5cc) — URGENT: 18 stand-down events logged (3 on 5/11 alone). Cron fires at UTC 0,1,2 = 8/9/10 PM EDT, causing tomorrow's recurring tasks to appear eligible tonight. Fix: change cron schedule to 0 11-23 * * * — awaiting user approval.
  • Daily Exec Agenda — {stamp} f-string bug: build_agenda.py has an unformatted placeholder {stamp} that appears in the output. Permanent fix needed in the script.
  • Combined Daily Brief — context-slug builder regression: Builder generates incomplete context slugs. Compliance Audit auto-remediates nightly. Root cause in the builder needs a code fix to prevent recurrence.
  • Skill-loading failure in cron subagents: Background cron subagents cannot load space skills (changeis-style, vm2-wrapup). load_skill returns "No space skill found" in background context. Workaround: parent agent performs any skill-dependent operations. Root cause: platform limitation in background cron execution context.
  • Todoist connector DISCONNECTED since 5/6: todoist__pipedream connector is not available. All crons use direct REST API v1 with cursor pagination as primary path. Connector reconnection is optional — everything works via REST.
  • Portal.html thread filter: Thread aggregator pages work via client-side portal.html fetch. The portal itself does not yet have a Thread dropdown filter UI alongside the Type pills.
  • Thread aggregator pages — 4 of 9 threads without pages: ydc-golf, exec-comp, apex-framework, and wsr-madison have thread tags in use but no aggregator page yet.
  • Outlook send_email subagent failure: Occasionally a subagent's send_email call fails silently. Pattern: parent session detects the failure and retries via call_external_tool directly.
  • Gmail dedup dual-write: Both gmail_intake/state.json (primary) and 6a1f1c3b/state.json (fallback) must be written after every intake run. Divergence causes missed or duplicate email processing.
  • Legacy vm2-opp and v2-done labels: These Todoist labels are retained but no longer applied by any active cron. Should be cleaned up or formally deprecated.

Key Rules (What VM2 Always Does)
  • Every deliverable → pushed to GitHub as plain HTML (no encryption)
  • Every deliverable → logged in portal.html (the Deliverables Index)
  • Every deliverable → includes context slug with cost_estimate for follow-up continuity
  • Every deliverable → shows Created and Revised dates in the header
  • Railway basic auth protects everything server-side (Tier 1)
  • portal.html = the index. index.html = redirect only. Never edit index.html.
  • All actions pre-approved — no confirmation prompts needed
  • Integrity audit runs twice daily (9 AM + 5 PM) as safety net
  • Deep dives are email-only — no automated dives, no credits spent on opportunities you didn't request (v2.2+)
  • SAM monitor uses authenticated api.sam.gov with SQLite persistence (v2.2+)
  • Thread tags are optional and additive — all existing deliverables work without them (v2.3)
  • Auto-Helper uses full task pool — direct REST API cursor pagination reaches all ~700 Todoist tasks (v2.3)
  • Public sharing uses Tier 2 GitHub Pages repos — dedicated public repo per shareable deliverable; Tier 1 row in portal.html stays unchanged (v2.4)
  • Todoist REST API is primary path — connector disconnected since 5/6; reconnection optional (v2.4)
  • Compliance Audit slug retrofits — when audit detects missing slugs, inline retrofit via inject_slug.py template (v2.4)
