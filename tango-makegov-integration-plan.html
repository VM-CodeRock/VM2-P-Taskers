Executive Summary
Bottom Line: Tango by MakeGov fills the most critical gap in the Changeis BD pipeline — historical award intelligence and incumbent detection — that raw SAM.gov cannot provide. The recommended path is a lightweight Railway microservice wrapping the Tango REST API, integrated into the existing SQLite/RAG stack. Build Phase 1-3 first; expect 3-6 week total effort at ~$25-100/month in Tango API credits depending on tier purchased.

Varun has purchased Tango API credits. Tango by MakeGov is a unified REST API that normalizes data from FPDS, USAspending, SAM.gov, Grants.gov, and 11 federal agency procurement forecast feeds into a single, developer-friendly interface. Unlike raw SAM.gov or direct FPDS queries, Tango delivers cleaned, deduplicated contract awards with vendor entity profiles, historical spending metrics, subaward data, and procurement forecasts — all queryable via a single API key.

The existing Changeis pipeline (SAM.gov monitor with 2,726-opp SQLite, APFS/DHS monitor, Combined Daily Brief, and deep-dive VM2 skill) handles opportunity discovery well but lacks competitive intelligence depth. Tango unlocks 7 high-value use cases, the top 3 to build first being: (1) Incumbent Intelligence Lookup, (2) Vendor Competitive Matrix for Capture Briefs, and (3) Procurement Forecast Early Warning.

Key Technical Features

  • Response Shaping: shape= parameter selects specific fields and expands nested objects (recipient, office, transactions) in a single request — reduces payload size for credit efficiency.
  • Full-text search across opportunity titles and descriptions — semantic-adjacent keyword search beyond basic NAICS filtering.
  • MCP Server at https://govcon.dev/mcp — 4 tools (resolve, search, search_opportunities, get_details) usable directly from Claude Desktop or Perplexity Computer via X-Tango-API-Key header.
  • Webhooks: Near-real-time push notifications for new awards, opportunities, entities, grants, and forecasts. Up to 1 (Free) / 3 (Micro) / 5 (Small) / 10 (Medium) / 25 (Large) subscriptions.
  • Attachment text extraction (Micro tier+): Solicitation PDF text extracted and indexed — enables RFI response research.
  • Entity monitoring (Large tier+): Track specific vendor entities for changes.
  • Python SDK (tango) and Node SDK (@makegov/tango-node) available.
  • Data freshness: SAM.gov opportunities every 20–60 min; contracts/awards typically within 24 hours of source updates.

Tango Cache Data Model (tango_awards.db)

-- Core award cache
CREATE TABLE tango_contracts (
  piid          TEXT PRIMARY KEY,
  vendor_uei    TEXT NOT NULL,
  vendor_name   TEXT,
  agency_code   TEXT,
  agency_name   TEXT,
  office_name   TEXT,
  naics         TEXT,
  psc           TEXT,
  award_amount  REAL,
  period_start  DATE,
  period_end    DATE,
  set_aside     TEXT,
  description   TEXT,
  fetched_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Vendor entity cache
CREATE TABLE tango_entities (
  uei           TEXT PRIMARY KEY,
  name          TEXT,
  business_types TEXT,  -- JSON array
  naics_primary  TEXT,
  state          TEXT,
  total_obligated REAL,
  contract_count  INTEGER,
  fetched_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Procurement forecast cache
CREATE TABLE tango_forecasts (
  forecast_id   TEXT PRIMARY KEY,
  agency_code   TEXT,
  agency_name   TEXT,
  title         TEXT,
  naics         TEXT,
  estimated_value REAL,
  anticipated_date DATE,
  set_aside     TEXT,
  description   TEXT,
  fetched_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Dedup/credit tracking
CREATE TABLE tango_api_calls (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  endpoint      TEXT,
  params_hash   TEXT,
  called_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  response_code INTEGER,
  credits_used  INTEGER DEFAULT 1
);

-- Index for fast incumbent lookup
CREATE INDEX idx_contracts_naics_agency ON tango_contracts(naics, agency_code);
CREATE INDEX idx_contracts_period_end   ON tango_contracts(period_end);

API key storageTANGO_API_KEY env var in Railway; injected as api_credentials=["tango"] pattern for VM2 skillsConsistent with existing Changeis secret management
Webhook receiverNew Railway endpoint or extend existing daily-brief serviceTango webhooks push to a public HTTPS URL; Railway already handles this for the RAG service
Response shapingAlways use shape= param to request only needed fieldsReduces payload size; faster; more predictable JSON structure

5. Build Options: Perplexity Computer vs. Claude vs. Codex
Option A
Perplexity Computer
Pros:
  • VM2 skills already wired; no new infra
  • Existing cron, Todoist, Railway integration
  • HTML deliverable pipeline in place
  • Tango MCP server works natively in Claude-compatible clients
  • api_credentials pattern already exists for secrets
Cons:
  • Step budget limits complex multi-call workflows
  • No persistent Python process (cron must re-initialize)
  • Complex data transformations need pre-built scripts
Effort: Low (3–5 days)  |  Quality: High for skills/briefs, moderate for heavy ETL
Option B
Claude (Anthropic API / Claude.ai)
Pros:
  • Tango MCP server works natively with Claude Desktop
  • Natural language to complex multi-step queries
  • Excellent at synthesizing procurement data into prose
  • Claude Code can write and run Python wrapper scripts
Cons:
  • Separate context/cost from existing VM2 setup
  • No native Todoist/cron integration without additional wiring
  • API costs add up for frequent on-demand use
Effort: Medium (1–2 weeks)  |  Quality: Excellent for research synthesis; good for automation
Option C
Codex / OpenAI API
Pros:
  • Codex/o3 excels at writing production Python code
  • Tango MCP works via OpenAI Responses API
  • Best for building standalone Railway microservice
  • Strong at structured data transformation + SQL
Cons:
  • Codex is primarily a code-generation tool, not an orchestrator
  • Requires more setup; no existing Changeis integration
  • Less natural for producing HTML deliverables
Effort: Medium-High (2–3 weeks)  |  Quality: Excellent for the Railway microservice; lower for skill/brief UX
Recommended Path: Perplexity Computer as orchestrator + Codex for the Railway wrapper microservice. Use VM2 skills for on-demand lookups (incumbent, vendor matrix) triggered from Daily Brief buttons and Todoist tasks. Use Codex to write and deploy the tango-wrapper Railway service that handles caching, rate limiting, and forecast webhooks. This hybrid approach leverages what already works while filling the gaps with purpose-built infrastructure.
