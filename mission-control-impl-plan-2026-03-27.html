Section 1

Architecture & Tech Stack

Technology decisions for Mission Control, grounded in the existing VM2 infrastructure and zero-new-dependency philosophy.

Core Technology Decision

Recommendation: Static HTML + Vanilla JS, served via existing Railway nginx.

VM2 already has a production-grade setup at vm2-p-taskers-production.up.railway.app (nginx with basic auth) backed by the VM-CodeRock/VM2-P-Taskers GitHub repo. Mission Control should be a single HTML file — mission-control.html — that lives in that repo, just like every other VM2 deliverable. No Node server, no Python backend, no new Railway service.

Why static HTML wins here: Every existing VM2 deliverable is already a self-contained HTML file. Railway serves them reliably. Git is the deployment pipeline. No runtime, no CORS, no environment variables, no server to crash. The dashboard reads data via fetch() calls to co-located JSON files and the Todoist REST API. That's it.

When a Lightweight Server Would Make Sense (and Why It Doesn't Yet)

A Node/Python backend becomes worthwhile only when Mission Control needs to: (1) write back to JSON files from the browser, (2) proxy authenticated API calls to hide secrets, or (3) run scheduled aggregation jobs that produce rollup JSON. None of these are Phase 1 requirements. If Phase 3 approvals require server-side state writes, revisit adding a lightweight Express endpoint as a second Railway service — but cross that bridge then.

Integration Points

1. Deliverables from portal.html

// Fetch portal.html, parse table rows into structured objects
async function loadDeliverables() {
  const res = await fetch('/portal.html');
  const html = await res.text();
  const doc = new DOMParser().parseFromString(html, 'text/html');
  const rows = doc.querySelectorAll('#tableBody tr');
  return Array.from(rows).map(tr => ({
    date: tr.dataset.date,
    title: tr.querySelector('.col-title a')?.textContent?.trim(),
    type: tr.dataset.type,
    url: tr.querySelector('.col-link a')?.getAttribute('href')
  })).filter(d => d.title);
}

2. Cron Health from state.json files

// Read each cron's state.json; compute health status
async function loadCronHealth(cronConfig) {
  const results = await Promise.allSettled(
    cronConfig.map(async cron => {
      const res = await fetch(cron.state_file);
      const state = await res.json();
      const lastRun = new Date(state.last_run || state.last_run_utc);
      const ageMinutes = (Date.now() - lastRun) / 60000;
      return {
        ...cron,
        state,
        lastRun,
        status: ageMinutes < 90 ? 'healthy' : ageMinutes < 240 ? 'warning' : 'stale'
      };
    })
  );
  return results.map(r => r.status === 'fulfilled' ? r.value : { error: r.reason });
}

3. Todoist Tasks via REST API

// Todoist API v2 — fetch active VM2 tasks
async function loadTodoistTasks(config) {
  const res = await fetch(
    `https://api.todoist.com/rest/v2/tasks?label=${encodeURIComponent('VM2')}`,
    { headers: { Authorization: `Bearer ${config.todoist.api_token}` } }
  );
  const tasks = await res.json();
  const today = new Date().toISOString().split('T')[0];
  return {
    overdue: tasks.filter(t => t.due?.date < today),
    dueToday: tasks.filter(t => t.due?.date === today),
    upcoming: tasks.filter(t => !t.due || t.due.date > today)
  };
}
Security note on the Todoist API token: Storing the token in _system/mc-config.json means it is served as a plain file from Railway (behind basic auth). This is acceptable for a personal dashboard behind basic auth, but note that anyone with the Railway credentials can read the file. For Phase 3, move the token server-side or use a Railway environment variable with a proxy endpoint.

Phase 2 — Todoist + Deliverables Integration

Deepen the integrations: richer Todoist views, smarter deliverable parsing, full-text search, and a live activity feed from git history.

Todoist Integration — Richer Task Views

Phase 1 shows tasks by label. Phase 2 adds project grouping, sub-tasks, priority flags, and a Kanban column view. The Kanban uses Todoist sections as column definitions:

  • Backlog: Tasks with no due date or due >3 days out
  • In Progress: Tasks due within 3 days, or manually flagged with a #in-progress label
  • Needs Approval: Tasks with a #needs-approval Todoist label (set by cron agents for gated actions)
  • Done (recent): Completed tasks from the last 7 days via GET /tasks?filter=completed&since=...

Deliverables — Enhanced Parsing

Phase 1 reads the basic table row. Phase 2 extracts richer metadata by fetching the individual deliverable HTML files and parsing their <meta> tags and first paragraph (for a summary snippet). This requires adding standard meta tags to new deliverables going forward:

<meta name="vm2-type" content="Architecture">
<meta name="vm2-tags" content="Mission Control, Dashboard, Planning">
<meta name="vm2-summary" content="Implementation plan for the VM2 Mission Control dashboard.">

The deliverables library then supports full-text search across titles, tags, and summaries — not just the table row text.

Search Implementation

Client-side fuzzy search using a minimal implementation (no library required for <500 records). Index is built once on page load from the deliverables array:

function searchDeliverables(query, deliverables) {
  const q = query.toLowerCase().trim();
  if (!q) return deliverables;
  return deliverables.filter(d =>
    d.title?.toLowerCase().includes(q) ||
    d.type?.toLowerCase().includes(q) ||
    d.tags?.some(t => t.toLowerCase().includes(q)) ||
    d.summary?.toLowerCase().includes(q)
  );
}

Activity Feed from Git Log

A bash script runs as a GitHub Actions workflow (or locally before push) to generate _system/activity.json. The workflow triggers on every push to main:

# .github/workflows/gen-activity.yml
name: Generate activity.json
on: [push]
jobs:
  gen:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
        with: { fetch-depth: 30 }
      - name: Generate activity.json
        run: |
          python3 _system/gen_activity.py > _system/activity.json
          git config user.email "vm2.viru@gmail.com"
          git config user.name "VM2 Computer"
          git add _system/activity.json
          git diff --staged --quiet || git commit -m "chore: update activity.json"
          git push

The gen_activity.py script outputs the last 30 commits as structured JSON with file counts, message, and date — displayed in the Dashboard as a "Recent Activity" feed.

Section 4

Phase 3 — Governance Layer

Approval queues, audit trails, and permission profiles — making irreversible actions visible and gated before execution.

The Core Problem Phase 3 Solves

VM2 agents currently execute irreversible actions (send email, post to external services, create calendar events, make purchases) within the same session that detects the need. There is no pause-and-verify step. Phase 3 introduces a structured approval workflow that inserts a human checkpoint before any flagged action executes.

Approval Queue Design

When a cron or agent needs approval, it writes a pending approval record to _system/mc-state.json and stops. Mission Control displays the queue. Varun clicks Approve or Reject. A webhook (or the next cron run) reads the decision and acts.

Implementation: How Agents Flag for Approval

In Phase 3, cron agents are updated to call a shared function before executing flagged actions:

# Pseudocode for cron agent
def execute_action(action_type, payload):
    risk = PERMISSION_MAP[action_type]['risk']
    if risk in ('medium', 'high'):
        write_approval_request(action_type, payload)  # → mc-state.json
        return "PENDING_APPROVAL"
    else:
        # Execute immediately, log to audit trail
        result = execute(payload)
        append_audit_log(action_type, payload, result)
        return result
Phase 3 requires a minimal server component. The browser cannot write JSON files to the Railway filesystem. The cleanest approach: add a tiny Node.js Express endpoint (10 lines) as a second Railway service that accepts POST /approve and POST /reject and writes to mc-state.json. All read operations remain static.
