01 System Overview & Architecture

What is VM2?

VM2 ("Varun Malhotra 2") is an automated Chief-of-Staff AI system built on Perplexity Computer for Varun Malhotra, President of Changeis Inc. It runs 14 scheduled tasks that automate:

  • Daily Executive Agenda — Morning briefing with calendar, email triage, and task priorities
  • Federal Opportunity Intelligence — Monitoring DHS APFS and SAM.gov for contract opportunities, RAG-scoring them against Changeis past performance, and generating deep-dive analyses
  • Task Automation — Executing tasks from Todoist (VM2 label) and Gmail (vm2.viru@gmail.com)
  • Compliance & Governance — Nightly audits, weekly digests, monthly integrity checks, deliverable integrity audits
  • Business Development — YPO introduction tracking, Todoist auto-helper suggestions, MAPS amendment monitoring

08e Gmail Attachment Retrieval (v2.5) NEW v2.5

Canonical three-step exhaust-order for retrieving email attachments via the Gmail connector (gcal source). Confirmed on BAFM RFP Analysis (PDF + xlsx) and WashExec Follow-Up (PNG from reply image).

Exhaust-Order (Always Follow in Sequence)

  1. get_email_attachments(email_id) — Fastest method. Returns attachment list with IDs and file metadata. Use this first for all emails with suspected attachments.
  2. read_email(id, include_attachments=true) — If step 1 fails or returns empty list. Reads the full email including inline attachments and embedded images.
  3. download_attachment(id) — Final fallback. Use when steps 1 and 2 both fail or return no usable attachment data. Requires attachment ID from a prior step or from the email headers.

Important: Never Skip Steps

Always start with step 1. Do not jump directly to step 2 or 3 even if the email subject suggests attachments are unusual. The exhaust-order is the canonical path. Document which step succeeded in the deliverable context slug.

Special Case: Reply-Image Attachments

When the user replies to an email with an image (e.g., screenshot of requirements, photo of whiteboard), the image appears as an inline attachment on the reply thread. Step 1 may return empty for inline images; proceed to step 2 with include_attachments=true, which captures inline content. OCR the image using fetch_url or image analysis to extract text content.

08f SAM Monitor Hardening (v2.6) NEW v2.6

Patch applied to /home/user/workspace/data/sam/daily_monitor.py on 2026-05-15 to prevent SQLite-lock errors from concurrent cron executions. The root cause was a concurrent-run scenario that triggered a SQLite lock escalation at 9:18 AM EDT on 5/15.

Root Cause (5/15/2026 9:18 AM)

A second invocation of daily_monitor.py started before the first completed. Both processes opened sam.sqlite in default journal mode (DELETE), causing lock contention and an unhandled OperationalError: database is locked that escalated to the foreground session.

Hardening Changes (v2.6)

WAL journal modePRAGMA journal_mode=WAL on every connection openAllows concurrent readers; reduces lock contention between writer and readers
Connection timeoutsqlite3.connect(database, timeout=30.0)Python-level wait up to 30 seconds before raising OperationalError
busy_timeout PRAGMAPRAGMA busy_timeout=30000 (30,000 ms)SQLite-level wait; handles lock at the C library layer before Python sees it
Lockfile/home/user/workspace/data/sam/.daily_monitor.lockProcess-level mutex: prevents second invocation from starting

Lockfile Procedure (Pseudocode)

LOCK_FILE = "/home/user/workspace/data/sam/.daily_monitor.lock"

def acquire_lock():
    if os.path.exists(LOCK_FILE):
        with open(LOCK_FILE) as f:
            pid = int(f.read().strip())
        try:
            os.kill(pid, 0)  # Check if process alive
            print(f"Another run in progress (PID {pid}). Exiting.")
            sys.exit(0)
        except OSError:
            print(f"Stale lockfile (PID {pid} dead). Removing.")
            os.remove(LOCK_FILE)
    with open(LOCK_FILE, 'w') as f:
        f.write(str(os.getpid()))

def release_lock():
    if os.path.exists(LOCK_FILE):
        os.remove(LOCK_FILE)

# Main flow:
try:
    acquire_lock()
    conn = sqlite3.connect(DB_PATH, timeout=30.0)
    conn.execute("PRAGMA journal_mode=WAL")
    conn.execute("PRAGMA busy_timeout=30000")
    # ... run logic ...
finally:
    release_lock()

08g SAM Stuck-Run Cleanup Procedure (v2.6) NEW v2.6

When the SAM.gov daily_monitor.py is killed mid-run (e.g., workspace reset, OOM, manual interrupt), it may leave a row in the runs table with status='in_progress'. The next invocation checks for in-progress runs and may refuse to start or behave unexpectedly.

Symptom

SAM monitor fails to produce output or skips the run with message like "Run already in progress since [timestamp]" even though no process is running. Lockfile is absent (or stale), but sam.sqlite runs table has an orphaned status='in_progress' row.

Cleanup Procedure

  1. Verify lockfile is absent or stale: 
    ls -la /home/user/workspace/data/sam/.daily_monitor.lock 2>/dev/null || echo "No lockfile"
  2. Check for orphaned in-progress run: 
    sqlite3 /home/user/workspace/data/sam/sam.sqlite \
  "SELECT id, start_time, status FROM runs WHERE status='in_progress';"
  3. If orphaned row found and no process is running: mark as failed: 
    sqlite3 /home/user/workspace/data/sam/sam.sqlite \
  "UPDATE runs SET status='failed', end_time=datetime('now')
   WHERE status='in_progress' AND start_time < datetime('now','-2 hours');"
  4. Remove stale lockfile if present: 
    rm -f /home/user/workspace/data/sam/.daily_monitor.lock
  5. Re-run the monitor: 
    python3 /home/user/workspace/data/sam/daily_monitor.py

Safety Condition

Only mark a run as failed if it has been in-progress for more than 2 hours (the start_time < datetime('now','-2 hours') clause). A run started recently might still be legitimately running in a background process. If in doubt, wait 2 hours before cleaning up.
