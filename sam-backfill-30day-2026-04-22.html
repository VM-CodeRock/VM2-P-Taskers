SAM.gov 30-Day Backfill & Database Persistence

Created: 2026-04-22
Executive Summary

The old SAM.gov daily monitor had been returning zero opportunities for 18 consecutive business days because it called the unauthenticated sam.gov/api/prod/sgs/v1/search endpoint whose schema had changed. Today we rebuilt the pipeline end-to-end:

  • Switched to the authenticated api.sam.gov/opportunities/v2/search endpoint with your API key
  • Built a persistent SQLite database at /home/user/workspace/data/sam/sam.sqlite as the single source of truth
  • Backfilled 30 days across 49 NAICS codes in the 541xxx family plus 518210, 561110, 611430
  • Applied Changeis RAG Engine scoring to every record
  • Rewrote cron 8f2a4ac1 to use the DB — every run is now idempotent and resumable

What Changed

Before

  • Unauthenticated endpoint returning 0 hits silently
  • JSON file seen_notice_ids.json as the only dedup mechanism
  • No history — only knew what was "new this run"
  • If a run failed, state was corrupt or partial
  • No RAG scoring before filtering

After

  • Authenticated api.sam.gov with retry/backoff
  • SQLite DB: 3 tables (opportunities, runs, opp_run_seen)
  • Every opp has first_seen_run_id, last_seen_run_id, full raw_json
  • Crash-safe: transactional upserts, re-runs are idempotent
  • RAG score attached to every record

Database Architecture

Three tables. Every notice_id ever seen is persisted. A failed run can be re-run and will not double-process.

TableRowsPurpose
opportunities1,45250 fields per opp: full API response + RAG score + lifecycle + workflow flags (deep_dive_requested, is_skipped, user_flagged)
runs2Every cron run with window, totals, errors, duration
opp_run_seenjunctionWhich opps were seen in which run, with is_new / is_updated flags

Indices on notice_id (PK), posted_date, rag_classification, active. Schema file and scripts live under /home/user/workspace/data/sam/.

How the Daily Monitor Works Now
  1. Cron 8f2a4ac1 fires weekdays at 9:00 AM EDT
  2. Runs python3 /home/user/workspace/data/sam/daily_monitor.py
  3. Queries api.sam.gov across 52 priority NAICS for the last 7 days
  4. Upserts into the DB — duplicates are transparent no-ops
  5. Calls RAG Engine for any truly NEW opps (score + top matches)
  6. Writes sam_daily_YYYYMMDDTHHMMSSZ.json for the Combined Daily Brief cron to pick up
  7. Sanity alarm fires if zero hits come back across all NAICS
Session Context

Original Instructions

"Pull all 541* opportunities for the last 30 days and reconcile against what we had already pulled. Figure out some way to make sure we keep a database so we can pick up where we left off if things break."

Approach

Built SQLite DB (3 tables, 50 opp fields, indices). Pulled api.sam.gov with authenticated key across 49 NAICS codes in the 541xxx family + 3 non-541 priorities (518210, 561110, 611430) using 10-day slices over the last 30 days. Upserted 1,452 unique opportunities. Applied RAG Engine scoring to every record. Updated cron 8f2a4ac1 to use the DB as single source of truth.
