1. System Overview

VM2 is a fully automated AI task pipeline built on Perplexity Computer. It receives work instructions via email, direct chat, or Todoist, executes them through AI agents, publishes HTML deliverables to a password-protected Railway web portal, and backs up everything to Dropbox and GitHub automatically. As of May 13, 2026, the repo contains 786 deliverables across 14 active scheduled crons.

2. Component Architecture

2.1 Portal & Index

Rule: portal.html is the sole deliverables index. index.html is an immutable ~229-byte redirect stub pointing to portal.html. No cron or session may modify index.html content. The nightly compliance audit enforces this.

Portal columns (v4.1+): portal.html has 5 columns plus optional thread tag attribute:

Column / AttributeClass / AttrSourceDescription
Createdcol-datedata-dateDate the deliverable was first published
Titlecol-title—Linked title opening the deliverable; may include secondary public share link in notes
Typecol-typedata-typeBadge-styled category (Deep Dive, Monitoring, Brief, etc.)
Updatedcol-updateddata-updatedDate of last revision (same as Created if v1)
Linkcol-link—Arrow icon linking to the deliverable
Thread Tag v4.3data-threadThread slugOptional attribute on <tr> — links deliverable to a cross-deliverable thread. Empty string if no thread.

Default sort: descending by data-updated so the most recently revised deliverables appear first.

Public Share convention (v4.4): When a deliverable has a Tier 2 public share URL, the original auth-protected row stays in portal.html unchanged; the public share URL is added as a secondary link in the title or notes column.

2.4.2 Gmail Attachment Retrieval (v4.5 · confirmed pattern)

When a task email contains attachments (PDFs, spreadsheets, images), use the following exhaust-order to retrieve them:

  1. Step 1: get_email_attachments(email_id) — returns attachment list with IDs
  2. Step 2 (if step 1 returns empty): read_email(email_id, include_attachments=true) — sometimes surfaces attachments not in the attachment list
  3. Step 3 (for specific file download): download_attachment(attachment_id) — download by ID once obtained from steps 1 or 2
Confirmed on 5/13: Used for BAFM PDF/xlsx (RFP analysis) and WashExec PNG screenshot (follow-up outreach). This three-step exhaust-order is now the canonical pattern for all Gmail attachment retrieval in VM2 subagents.

4.2 Sync Mechanism

The nightly vm2-wrapup skill (v2.1+) performs a full diff sync:

  1. Lists all .html files in /VM2-main-folder/VM2-P/Taskers/ recursively
  2. Lists all deliverable HTML files in the GitHub repo
  3. Computes the diff (GitHub − Dropbox)
  4. Uploads all missing files to the correct categorized subfolder
  5. Reports N synced, N already present in the compliance report

4.3 Known Issue — Duplicate-on-Overwrite (v4.5)

Dropbox connector "(1)" duplicate bug: When a file is uploaded to a path that already exists, the Dropbox connector creates a "(1)" suffix duplicate instead of overwriting. This causes deliverables to accumulate duplicate entries (e.g., file.html and file (1).html). The 5/13 audit detected 8–10 missing deliverables that were auto-remediated in both the 1 PM and 9 PM audit runs. Fix needed: investigate destination path normalization — ensure uploads use mode: overwrite in the Dropbox API call rather than relying on connector defaults.

5. GitHub & Railway Hosting

5.1 Push Workflow

# Standard deliverable push
git add <filename>.html portal.html
git commit -m "Deliverable: <slug> · VM2 · YYYY-MM-DD"
git pull --rebase origin main   # resolve any concurrent pushes
git push origin HEAD
# Railway auto-deploys within ~60 seconds

5.2 Non-Fast-Forward Recovery (v4.5)

Standard resolution for non-fast-forward push failures: git pull --rebase origin main then git push origin HEAD. Never use force-push (git push --force) — this risks overwriting concurrent deliverables. The Combined Daily Brief cron hit this on 5/13 (~10:08 AM); parent agent resolved via rebase.

5.3 HTML Requirements

  • All HTML must be plain HTML — no StaticCrypt encryption
  • Railway nginx basic auth protects all pages server-side
  • Deliverables should include a "Continue This Work" footer panel with context for follow-up sessions
  • Context slug at write-time (v4.5): The Combined Daily Brief cron MUST add the vm2-context-data slug at write-time, not via post-hoc retrofit. Slug retrofit is a recovery procedure, not a standard flow.

6. Thread Tag Mechanism v4.3

Shipped May 5, 2026. The Thread Tag Mechanism binds related deliverables into named threads — without changing how any existing file or workflow operates. It is fully additive.

6.1 Design Principle

Threads and versions are orthogonal. Versions chain a single deliverable (v1 → v2 → v3). Threads aggregate multiple different deliverables into a named topic. A deliverable can have both simultaneously.

7. Public Sharing Pattern v4.4

Established May 11, 2026 during the Mahjong Tutor follow-up. The two-tier model separates BD/proprietary deliverables (auth-required) from publicly shareable deliverables (e.g., apps, tools, family/external-audience content).

7.1 Two-Tier Publishing Model

TierHostingAuthScope
Tier 1 (Auth-protected)Railway at vm2-p-taskers-production.up.railway.appnginx basic auth (vm2 / 97harry23!)All BD deliverables, proprietary analysis, system docs — for Changeis team
Tier 2 (Public)Dedicated GitHub Pages repo per deliverable: vm-coderock/{slug}-shareNone — publicUser explicitly requests public URL OR deliverable is for non-Changeis audience (family, external collaborators)

7.2 When to Use Tier 2

  • User explicitly requests a publicly shareable URL for a deliverable
  • Deliverable is intended for non-Changeis audience (e.g., family, external collaborators, public demos)
  • Examples: Mahjong Tutor, game/tool deliverables, any deliverable to be shared externally

7.3 Implementation

# Create public share repo
gh repo create vm-coderock/{slug}-share --public

# Push HTML as index.html
git init {slug}-share
cp {deliverable}.html {slug}-share/index.html
cd {slug}-share && git add . && git commit -m "Initial"
git remote add origin https://github.com/vm-coderock/{slug}-share.git
git push -u origin main

# Enable GitHub Pages
gh api -X POST /repos/vm-coderock/{slug}-share/pages \
  -f source[branch]=main -f source[path]=/

# Result: https://vm-coderock.github.io/{slug}-share/

The original auth-protected row in portal.html stays unchanged. The public share URL is added as a secondary link in the title or notes column — e.g., [Public ↑](https://vm-coderock.github.io/{slug}-share/).
