1. System Overview

VM2 is an automated task pipeline that receives work instructions via email, executes them through Perplexity Computer AI agents, and publishes HTML deliverables to a password-protected web portal. The system runs continuously with hourly email monitoring, nightly compliance audits, and weekly digests.

Key Architecture Decision

As of March 2026, all client-side StaticCrypt encryption has been removed. Password protection is handled server-side by nginx basic auth on Railway. This eliminates the double-login problem and simplifies the entire pipeline.

2. Component Architecture

2.1 Railway Hosting & Authentication

The deliverables portal is hosted on Railway using a custom Docker container built on nginx:alpine. Authentication is handled entirely server-side.

PropertyValue
PlatformRailway (us-west2)
BuilderDockerfile
Base Imagenginx:alpine
Port8080 (internal), mapped by Railway
Auth Methodnginx auth_basic with .htpasswd
Usernamevm2 (env var: VM2_AUTH_USERNAME)
PasswordStored in VM2_AUTH_PASSWORD env var
Auto-deployYes — pushes to main trigger rebuild

How Auth Works at Runtime

  1. entrypoint.sh runs at container startup
  2. Reads VM2_AUTH_USERNAME and VM2_AUTH_PASSWORD from environment
  3. Generates /etc/nginx/.htpasswd using openssl passwd -apr1
  4. Starts nginx with auth_basic pointing to that file
  5. Every request requires HTTP Basic Auth — one login, no second prompt

Dockerfile

FROM nginx:alpine
RUN apk add --no-cache openssl
COPY nginx.conf /etc/nginx/conf.d/default.conf
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
COPY . /usr/share/nginx/html/
RUN rm -f /usr/share/nginx/html/Dockerfile \
    /usr/share/nginx/html/nginx.conf \
    /usr/share/nginx/html/entrypoint.sh \
    /usr/share/nginx/html/.staticrypt.json \
    /usr/share/nginx/html/.gitignore
EXPOSE 8080
ENTRYPOINT ["/entrypoint.sh"]

nginx.conf

server {
    listen 8080;
    server_name _;
    root /usr/share/nginx/html;
    index index.html;

    auth_basic "VM2 Deliverables";
    auth_basic_user_file /etc/nginx/.htpasswd;

    location / {
        try_files $uri $uri/ =404;
    }

    location ~* \.(css|js|png|jpg|gif|ico|svg|woff|woff2)$ {
        expires 7d;
        add_header Cache-Control "public, immutable";
    }

    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "SAMEORIGIN" always;

    gzip on;
    gzip_types text/html text/css application/javascript text/plain;
    gzip_min_length 256;
}

entrypoint.sh

#!/bin/sh
USERNAME="${VM2_AUTH_USERNAME:-vm2}"
PASSWORD="${VM2_AUTH_PASSWORD:-97harry23!}"
echo "${USERNAME}:$(openssl passwd -apr1 "${PASSWORD}")" > /etc/nginx/.htpasswd
echo "VM2 auth configured for user: ${USERNAME}"
exec nginx -g "daemon off;"

2.3 Gmail Task Pipeline

The Gmail pipeline is the primary task intake mechanism. Varun sends an email to vm2.viru@gmail.com with "VM2" in the subject or body, and the system automatically executes the instructions.

Email Processing Flow

  1. Hourly cron fires (weekdays, 8:29 AM to 7:29 PM AST)
  2. Searches Gmail for emails containing "VM2" without the vm2-done label
  3. Checks state.json for previously processed email IDs (dedup)
  4. For each new task email:
    • Parses subject (task title) and body (instructions)
    • Derives a logical task name (3-6 words, title case)
    • Spawns a dedicated subagent via run_subagent
  5. Each subagent independently:
    • Executes the task per email instructions
    • Creates an HTML deliverable in Changeis brand style
    • Pushes plain HTML to GitHub
    • Updates index.html
    • Saves a copy to Dropbox
    • Creates/completes a Todoist task
  6. Replies to the original email thread with a completion summary
  7. Updates tracking state for deduplication
Deduplication Mechanism

The Gmail connector cannot apply labels programmatically. Deduplication relies on a tracking file at /home/user/workspace/cron_tracking/aa619acf/state.json that stores processed email IDs. The vm2-done label in Gmail serves as a secondary visual indicator only.

2.7 Index Generator

Why StaticCrypt Was Removed

The original system used StaticCrypt to encrypt each HTML file client-side, requiring a password to decrypt in the browser. This created a double-login problem (HTTP basic auth + StaticCrypt password prompt) and added significant complexity to the pipeline. With Railway handling auth server-side, client-side encryption is redundant and was eliminated.

11. Migration History

March 25, 2026 — StaticCrypt to Railway Migration

The VM2 system was migrated from client-side StaticCrypt encryption + GitHub Pages to server-side nginx Basic Auth on Railway.

What Changed

BeforeAfter
GitHub Pages hostingRailway Docker hosting
StaticCrypt encryption (per-file)nginx Basic Auth (server-side)
Double login (HTTP + StaticCrypt)Single login (HTTP Basic Auth only)
3 different passwords across files1 unified password
Client-side decryption in browserPlain HTML served directly
Complex cron encryption stepSimple git push

Migration Steps Executed

  1. Phase 1: Decrypted all 95 encrypted HTML files using custom Node.js decryptor (PBKDF2 3-round with UTF-8 salt). Passwords used: 97harry23! (93 files), jungpura23 (~15 files), changeis2026 (2 files). Two files were double-encrypted.
  2. Phase 2: Auto-generated new index.html with build_index.py — 16 type categories, search, pills, date range, sort, dark/light toggle.
  3. Phase 3: Created Dockerfile, nginx.conf, entrypoint.sh for Railway.
  4. Phase 4: Committed and pushed. Resolved merge conflict from concurrent cron push.
  5. Phase 5: Recreated all 6 crons with updated instructions (no encryption). Updated vm2-gmail-tasks skill v2→v3, vm2-wrapup skill v1→v2.
  6. Phase 6: Disabled GitHub Pages via API. Fixed post-deploy re-encryption from old cron.

StaticCrypt Decryption Algorithm (Historical Reference)

// 3-round PBKDF2 derivation — salt is UTF-8 string bytes, NOT hex-decoded
Round 1: PBKDF2(password, UTF8(salt), 1000 iterations, SHA-1)    → hex string
Round 2: PBKDF2(round1_hex, UTF8(salt), 14000 iterations, SHA-256) → hex string
Round 3: PBKDF2(round2_hex, UTF8(salt), 585000 iterations, SHA-256) → 32 bytes

// Then AES-256-CBC decrypt
iv  = encrypted_blob[0..16]
data = encrypted_blob[16..]
AES-256-CBC(key=round3, iv=iv, data=data) → decrypted HTML
